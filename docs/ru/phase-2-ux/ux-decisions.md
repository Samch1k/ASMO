# ASMO -- UX-решения

Этот документ фиксирует ключевые UX-решения, принятые для ASMO, рассмотренные альтернативы и обоснование каждого выбора.

---

## Таблица решений

| # | Решение | Выбор | Альтернативы | Обоснование |
|---|---------|-------|--------------|-------------|
| 1 | Тип интерфейса | Только CLI | CLI + Web UI; только Web; расширение VS Code | Интеграция в workflow разработчика, нулевые накладные расходы, скриптуемость, работа по SSH, без зависимости от браузера |
| 2 | Поддержка языков | Двуязычный EN/RU (автодетекция) | Только EN; полный i18n-фреймворк (i18next) | Основная команда двуязычна; детекция кириллицы тривиальна (regex-проверка); i18n-фреймворк избыточен для 2 языков |
| 3 | Формат вывода | Эмодзи-индикаторы + структурированный текст | Только текст; только JSON; TUI с ncurses | Эмодзи обеспечивают мгновенное визуальное восприятие; структурированный текст совместим с grep; `--json` для машинного потребления |
| 4 | Подход к конфигурации | 3-уровневый fallback (проект > пользователь > встроенные) | Один файл; только переменные окружения; только dotfile | Zero-config из коробки; проектные переопределения для команд; пользовательские для личных предпочтений; встроенные всегда доступны через npm |
| 5 | UX одобрения | YOLO-автообход при score < 30 | Всегда требовать одобрение; никогда не требовать; настраиваемый порог | Баланс: безопасность для сложных задач, скорость для тривиальных; порог настроен под реальное распределение сложности |
| 6 | Отображение ошибок | Категоризация (retryable/fatal/validation) | Общие сообщения; только коды ошибок; стек-трейсы | Действенная обратная связь -- пользователь сразу понимает, повторить попытку или исправить что-то |
| 7 | Отображение прогресса | Фазовые индикаторы | Только спиннер; потоковый вывод логов; тишина до завершения | Понятный прогресс без перегрузки; каждая фаза соответствует агенту, делая мультиагентный процесс прозрачным |
| 8 | Персонажи агентов | BMAD-агенты с именами и характерами | Общие идентификаторы (agent-1, agent-2); только роли | Запоминаемость; персонажи естественно обеспечивают принципы качества (Amelia блокирует при падающих тестах, Bob при размытости) |

## Детальное обоснование

### 1. Только CLI

ASMO нацелен на профессиональных разработчиков, работающих преимущественно в терминале. CLI-интерфейс:

- Интегрируется в существующие shell-процессы (`asmo run` так же естествен, как `git commit`)
- Полностью скриптуем и композируем с другими инструментами (`asmo suggest | jq`)
- Работает по SSH, в контейнерах и в CI/CD-пайплайнах
- Имеет нулевую стоимость запуска -- без браузера, Electron или привязки к портам
- Избавляет от бремени поддержки веб-фронтенда

Расширение для VS Code было рассмотрено, но отложено. CLI может быть обёрнут расширением позднее без изменения основной архитектуры.

### 2. Двуязычная автодетекция

Алгоритм прост: если строка ввода содержит кириллические символы (Unicode-диапазон `\u0400-\u04FF`), вывод на русском; иначе -- на английском. Этот подход:

- Не требует конфигурации от пользователя
- Не имеет ложных срабатываний (кириллица однозначна)
- Не подтягивает i18n-библиотеки (без `i18next`, файлов локалей, ключей перевода)
- Масштабируется под реальную потребность: два языка, не двадцать

При необходимости большего количества языков i18n-фреймворк можно добавить без нарушения существующей детекции.

### 3. Эмодзи + структурированный вывод

Эмодзи-индикаторы (галочки, спиннеры, символы статуса) обеспечивают мгновенный визуальный контекст без необходимости читать полные предложения. В сочетании со структурированным key-value выводом пользователи могут быстро сканировать результаты. Флаг `--json` предоставляет машиночитаемую альтернативу.

Чистый текст отвергнут из-за отсутствия визуальной иерархии. JSON-only отвергнут как недружелюбный для интерактивного использования.

### 4. Трёхуровневая конфигурация

Цепочка fallback (проект > пользователь > встроенные) означает:

- **Новые пользователи** получают работающую систему немедленно (встроенные шаблоны)
- **Отдельные пользователи** могут кастомизировать, не затрагивая команду (пользовательский уровень)
- **Команды** могут стандартизировать workflow по проектам (проектный уровень)
- **npm-пакетирование** работает естественно -- встроенные шаблоны поставляются с `@asmo/core`

Один файл конфигурации вынудил бы дублировать встроенные шаблоны. Только переменные окружения не могут выразить сложные определения workflow.

### 5. Порог YOLO-режима

Порог 30 (из 100) выбран на основе распределения сложности:

- Оценка 0--15: тривиальные (опечатки, однострочные изменения) -- всегда безопасно автоодобрять
- Оценка 16--29: низкая сложность (простые переименования, мелкие баг-фиксы) -- безопасно для автоодобрения
- Оценка 30+: средняя и выше -- человеческое суждение добавляет ценность

Порог настраивается по проектам через конфигурацию. Значение по умолчанию консервативно: автоодобряются только задачи, где надзор человека не добавляет ценности.

### 6. Категоризированные ошибки

Ошибки делятся на три категории:

| Категория | Пример | Действие пользователя |
|---|---|---|
| **Повторяемая** | Таймаут LLM, rate limit | Подождать и повторить (автоповтор встроен) |
| **Валидация** | Задача слишком короткая, невалидный UTF-8 | Исправить ввод |
| **Фатальная** | Отсутствует конфигурация, сработал CircuitBreaker | Исследовать и исправить корневую причину |

Общие сообщения вроде "Что-то пошло не так" не дают пользователю пути вперёд. Категоризированные ошибки делают каждый сбой действенным.

### 7. Фазовый прогресс

Каждый workflow разделён на фазы, и каждая фаза соответствует одному или нескольким агентам. Отображение прогресса как `Фаза 2/3: Реализация (developer) -- 5м 32с` даёт пользователю:

- Ясное понимание позиции в workflow (2 из 3 завершено)
- Какой агент работает (developer)
- Сколько времени занимает каждая фаза (для будущих оценок)

Потоковый вывод логов перегружает пользователя. Одиночный спиннер полностью скрывает прогресс. Фазовое отображение -- золотая середина.

### 8. BMAD-персонажи агентов

Именованные агенты с персональностями служат двум целям:

1. **Запоминаемость**: "Amelia заблокировала сборку" запоминается лучше, чем "QA-агент вернул ошибку"
2. **Обеспечение качества**: каждая персональность воплощает принцип:
   - **Amelia** (QA): не одобрит непротестированный код
   - **Winston** (Architect): настаивает на документации и ADR
   - **Bob** (Product Owner): отклоняет размытые требования
   - **John** (Developer): предпочитает прагматичные, работающие решения

Это не косметика -- характерные черты закодированы в промптах агентов и напрямую влияют на принятие решений при выполнении workflow.

## Будущие рассмотрения

| Пункт | Статус | Примечания |
|---|---|---|
| Расширение VS Code | Отложено | Обёртка CLI, изменения ядра не нужны |
| Веб-дашборд | Не планируется | Метрики можно экспортировать в существующие инструменты |
| Дополнительные языки | Не планируется | i18n-фреймворк можно добавить при необходимости |
| Настраиваемый порог YOLO | Планируется | По-проектное переопределение через конфигурацию |
| Интерактивный режим (`asmo shell`) | На рассмотрении | REPL для итеративного уточнения задач |
