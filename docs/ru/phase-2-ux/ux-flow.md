# ASMO -- UX-поток

Этот документ описывает сквозные потоки взаимодействия для каждого основного сценария использования. Каждый поток показывает, что происходит внутри системы и что видит пользователь на каждом этапе.

---

## Поток 1: Стандартное выполнение (`asmo run`)

Основной поток. Пользователь предоставляет описание задачи; ASMO анализирует его, выбирает workflow и выполняет через соответствующих агентов.

```
Ввод пользователя
    │
    v
┌─────────────────┐
│  Инициализация   │  Валидация ввода, загрузка конфигурации (3 уровня), инициализация LLM
└────────┬────────┘
         │         Пользователь видит: "Инициализация ASMO..."
         v
┌─────────────────┐
│     Анализ       │  Оценка задачи 0-100, классификация уровня, детекция ключевых слов
│    сложности     │  Определение домена (security, performance, bug и др.)
└────────┬────────┘
         │         Пользователь видит: "Анализ сложности задачи..."
         v
┌─────────────────┐
│     Выбор        │  Подбор лучшего workflow из 27 доступных
│    workflow       │  Формирование команды агентов по определению workflow
└────────┬────────┘
         │         Пользователь видит: резюме анализа (оценка, workflow, уверенность, агенты)
         v
┌─────────────────┐
│   Детекция фаз   │  Проверка существующих артефактов (тесты, спецификации, код)
│   (адаптивная)   │  Определение оптимальной стартовой фазы (пропуск завершённой работы)
└────────┬────────┘
         │         Пользователь видит: ничего (прозрачная оптимизация) или "Вход с фазы N"
         v
┌─────────────────┐
│   Одобрение      │  Если score >= 30: запрос одобрения у пользователя
│   (checkpoint)   │  Если score <  30: YOLO автообход (без запроса)
└────────┬────────┘
         │         Пользователь видит: запрос одобрения ИЛИ "Автоодобрено (YOLO)"
         v
┌─────────────────┐
│   Выполнение     │  Запуск агентов последовательно/параллельно по workflow
│    агентов       │  Каждый агент: LLM-вызов -> обработка -> артефакты
│     (цикл)       │  CircuitBreaker: повтор при сбое, прерывание при повторных сбоях
│                  │  ContextCascade: передача артефактов между агентами
└────────┬────────┘
         │         Пользователь видит: пофазный прогресс с именем агента и таймингом
         v
┌─────────────────┐
│    Сбор          │  Запись длительности, потребления токенов, успеха/неудачи
│    метрик        │  Передача в MetricsOptimizer для цикла обучения
└────────┬────────┘
         │
         v
┌─────────────────┐
│    Результат     │  Резюме: длительность, изменённые файлы, результаты тестов
└─────────────────┘
         │         Пользователь видит: финальное резюме с итогом
```

**Типичная длительность:** 2--15 минут в зависимости от сложности задачи и количества агентов.

## Поток 2: Быстрая рекомендация (`asmo suggest`)

Спроектирован для хуков, автоматизации и быстрых проверок. Возвращает JSON-рекомендацию без выполнения.

```
Ввод пользователя
    │
    v
┌─────────────────┐
│    Лёгкий        │  Быстрая оценка сложности (LLM или эвристики)
│    анализ        │  Детекция ключевых слов, расчёт оценки
└────────┬────────┘
         │
         v
┌─────────────────┐
│   JSON-вывод     │  { useAsmo, confidence, workflow, agents, complexity, reasoning }
└─────────────────┘
```

**Типичная длительность:** менее 3 секунд.

Этот поток намеренно минимален. Используется тот же ComplexityAnalyzer, но выполнение workflow полностью пропускается. JSON-вывод стабилен и подходит для пайплайнов с другими инструментами:

```bash
# Пример: интеграция с хуком
if asmo suggest "$TASK" | jq -r '.useAsmo'; then
  asmo run "$TASK"
fi
```

## Поток 3: Пробный запуск (`asmo run --dry-run`)

Полный конвейер анализа без выполнения. Показывает пользователю, что именно ASMO сделал бы.

```
Ввод пользователя
    │
    v
┌─────────────────┐
│  Полный анализ   │  Аналогично потоку 1: сложность, выбор workflow, детекция фаз
└────────┬────────┘
         │
         v
┌─────────────────┐
│  Отображение     │  Выбранный workflow, последовательность агентов, ожидаемые фазы
│  плана           │  Обнаруженные артефакты, стартовая фаза, статус одобрения
└────────┬────────┘
         │
         v
   ┌──────────┐
   │   СТОП   │  Агенты не запускаются. Файлы не модифицируются.
   └──────────┘
```

**Типичная длительность:** 5--15 секунд.

Пользователь видит тот же анализ, что и при реальном запуске, но с чётким маркером:

```
[DRY RUN] Агенты не были запущены. Используйте `asmo run` без --dry-run для выполнения.
```

## Поток 4: YOLO-режим (автоматическое срабатывание)

YOLO-режим -- не отдельная команда, а автоматическое поведение, срабатывающее при оценке сложности ниже порога (по умолчанию: 30).

```
Ввод пользователя
    │
    v
┌─────────────────┐
│     Анализ       │  Оценка задачи
└────────┬────────┘
         │
         v
    Score < 30?
    ┌────┴────┐
   ДА        НЕТ
    │         │
    v         v
┌────────┐  ┌──────────┐
│  Авто- │  │  Запрос   │
│ одобре-│  │  одобре-  │
│  ние   │  │   ния     │
│ (YOLO) │  │           │
└───┬────┘  └────┬─────┘
    │             │
    v             v
┌─────────────────┐
│   Выполнение     │  Стандартный цикл агентов
└────────┬────────┘
         v
┌─────────────────┐
│    Результат     │
└─────────────────┘
```

Пользователь видит `Автоодобрено (YOLO-режим, оценка: 22)` вместо интерактивного запроса. Это устраняет трение для тривиальных задач: исправление опечаток, простые переименования, однострочные изменения.

## Поток 5: Ручной выбор workflow (`asmo run --workflow`)

Обходит автоматический выбор workflow. Пользователь указывает, какой именно workflow запустить.

```
Ввод пользователя + --workflow <id>
    │
    v
┌─────────────────┐
│  Инициализация   │  Валидация ввода, загрузка конфигурации
└────────┬────────┘
         │
         v
┌─────────────────┐
│     Анализ       │  Всё равно выполняется (для метрик и YOLO-решения)
│    сложности     │  Результат выбора workflow ИГНОРИРУЕТСЯ
└────────┬────────┘
         │
         v
┌─────────────────┐
│  Использование   │  Загрузка workflow по ID, пропуск логики выбора
│  указанного WF   │  Проверка существования workflow
└────────┬────────┘
         │
         v
┌─────────────────┐
│  Стандартное     │  Далее как в потоке 1: одобрение -> выполнение -> результат
│  выполнение      │
└─────────────────┘
```

Пользователь видит в секции анализа:

```
Workflow: security_audit (ручной выбор, автовыбор пропущен)
```

Это полезно, когда пользователь знает лучше эвристики -- например, принудительный запуск `security_audit` на коде, который ASMO классифицировал бы как простой рефакторинг.

## Обработка ошибок

Все потоки используют единый паттерн отображения ошибок:

```
Повторяемые ошибки:       "Таймаут LLM-провайдера. Повтор (2/3)..."
Фатальные ошибки:         "Файл конфигурации не найден: ~/.asmo/config/workflows/custom.json"
Ошибки валидации:          "Задача слишком короткая (2 символа). Минимум: 3 символа."
Срабатывание CircuitBreaker: "Агент 'tester' упал 3 раза. Прерывание workflow."
```

Ошибки категоризированы, чтобы пользователь сразу понимал, стоит ли повторить попытку или нужно исправить что-то конкретное.
