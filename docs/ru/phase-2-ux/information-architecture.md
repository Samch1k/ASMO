# ASMO -- Информационная архитектура

## Дерево CLI-команд

```
asmo
├── run <task>                # Основная команда -- анализ + выполнение
│   ├── --verbose             # Детальный вывод (шаги агентов, LLM-вызовы, тайминги)
│   ├── --dry-run             # Полный анализ без выполнения
│   ├── --use-api             # Принудительный API-режим (требует ANTHROPIC_API_KEY)
│   ├── --no-llm              # Только эвристики (~65% точности, оффлайн)
│   └── --workflow <id>       # Ручной выбор workflow (пропуск автовыбора)
│
├── suggest <task>            # Быстрая рекомендация (JSON-вывод)
│   ├── --threshold <n>       # Порог сложности для решения useAsmo
│   └── --no-json             # Человекочитаемый вывод вместо JSON
│
├── analyze <task>            # Только анализ сложности
│   ├── --context <path>      # Контекст проекта для улучшения анализа
│   └── --json                # Машиночитаемый JSON-вывод
│
├── workflow [name]           # Список всех workflow или выполнение конкретного
│   ├── --task <task>         # Описание задачи (обязательно при выполнении)
│   ├── --dry-run             # План выполнения без запуска
│   └── --verbose             # Детальный вывод workflow
│
└── task <command>            # Управление задачами
    ├── create <title>        # Создать новую задачу
    ├── list                  # Список всех задач
    ├── show <id>             # Показать детали задачи
    ├── start <id>            # Отметить задачу как в работе
    ├── complete <id>         # Отметить задачу как завершённую
    └── fail <id>             # Отметить задачу как проваленную
```

## Иерархия конфигурации

ASMO разрешает конфигурацию через трёхуровневую цепочку fallback. Каждый уровень перекрывает нижестоящий.

### Уровень 1 -- Проектный (высший приоритет)

```
.cursor/config/
├── orchestration/
│   └── workflows/          # Проектные переопределения workflow
├── roles/                  # Проектные определения ролей
└── skills/                 # Проектные файлы навыков
```

Располагается в корне проекта. Настройки применяются только к текущему проекту и имеют наивысший приоритет. Полезно для команд, которым нужна проектная настройка workflow.

### Уровень 2 -- Пользовательский

```
~/.asmo/config/
├── workflows/              # Персональные кастомизации workflow
├── roles/                  # Персональные переопределения ролей
└── skills/                 # Персональные добавления навыков
```

Располагается в домашней директории пользователя. Настройки применяются ко всем проектам данного пользователя. Полезно для личных предпочтений и часто используемых кастомных workflow.

### Уровень 3 -- Встроенные шаблоны (низший приоритет, всегда доступны)

```
node_modules/@asmo/core/templates/
├── workflows/              # 27 определений workflow (JSON)
├── roles/                  # 5 JSON-файлов ролей, содержащих 25 ролей
├── skills/                 # 55 SKILL.md файлов
└── teams/                  # Определения составов команд
```

Поставляется с пакетом `@asmo/core`. Это встроенные шаблоны по умолчанию, обеспечивающие работу ASMO без конфигурации. Пользователь их не модифицирует.

### Порядок разрешения

```
Проект (.cursor/config/)  -->  Пользователь (~/.asmo/config/)  -->  Встроенные (node_modules/@asmo/core/templates/)
```

При загрузке workflow, роли или навыка ASMO проверяет каждый уровень по порядку и использует первое совпадение. Это означает, что проектный `bug_fix_workflow.json` перекроет встроенный шаблон, не затрагивая другие проекты.

## Поток данных

```
CLI-ввод (asmo run "задача")
  │
  ├── InputValidator          # Санитизация, проверка длины, UTF-8, контрольных символов
  │
  ├── ConfigLoader            # Загрузка workflows/roles/skills из 3-уровневой иерархии
  │     ├── Роли (JSON)
  │     ├── Навыки (SKILL.md)
  │     └── Воркфлоу (JSON-шаблоны)
  │
  ├── LLMProviderFactory      # Session -> API -> Heuristics (автоматический fallback)
  │
  ├── ComplexityAnalyzer      # Оценка 0-100, классификация по 5 уровням
  │
  ├── WorkflowSelector        # Подбор лучшего workflow для задачи
  │
  ├── PhaseDetector           # Определение фазы для старта (адаптивный вход)
  │
  ├── ApprovalCheckpoint      # YOLO-обход (<30) или одобрение пользователем
  │
  ├── WorkflowEngine          # Оркестрация выполнения агентов
  │     ├── AgentExecutor     # Запуск отдельных агентов
  │     ├── CircuitBreaker    # Отказоустойчивость на уровне агента
  │     └── ContextCascade    # Передача контекста между агентами
  │
  └── MetricsCollector        # Длительность, потребление токенов, успех/неудача
        └── MetricsOptimizer  # Цикл обучения для улучшения будущих решений
```

## Структура пакетов

```
packages/
├── core/                   # @asmo/core -- библиотека оркестрации
│   ├── src/
│   │   ├── agents/         # Реализации агентов
│   │   ├── workflows/      # WorkflowEngine, WorkflowSelector
│   │   ├── analysis/       # ComplexityAnalyzer, KeywordAnalyzer
│   │   ├── config/         # ConfigLoader, 3-уровневое разрешение
│   │   ├── llm/            # LLMProviderFactory, Session/API-провайдеры
│   │   └── utils/          # InputValidator, CircuitBreaker и др.
│   ├── templates/          # Встроенные шаблоны (workflows, roles, skills)
│   └── docs/               # Внутренняя техническая документация
│
└── cli/                    # @asmo/cli -- интерфейс командной строки
    └── bin/
        └── asmo.js         # Точка входа
```
