# Безопасность и приватность

## Валидация ввода

Весь пользовательский ввод валидируется перед входом в конвейер оркестрации. ASMO выполняет 10 проверок на каждом промпте задачи.

| # | Проверка | Действие |
|---|----------|----------|
| 1 | Проверка на пустоту/null | Отклонение с описательной ошибкой |
| 2 | Ограничения длины | Обеспечение диапазона 3--10 000 символов |
| 3 | Валидация кодировки UTF-8 | Отклонение невалидных байтовых последовательностей |
| 4 | Удаление контрольных символов | Очистка символов, способных повлиять на рендеринг терминала |
| 5 | Нормализация пробелов | Схлопывание избыточных пробелов, обрезка краёв |
| 6 | Санитизация null-байтов | Удаление null-байтов, способных обрезать строки |
| 7 | Защита от prompt injection | Обнаружение и нейтрализация типичных паттернов инъекций |
| 8 | Zod-валидация схем | Валидация всех структурированных вводов по Zod-схемам |
| 9 | Ролевые ограничения MCP | Обеспечение доступа агентов только к разрешённым MCP-серверам |
| 10 | Изоляция API-ключей | Проверка, что ключи берутся только из переменных окружения |

Валидация применяется в точке входа CLI до любых LLM-вызовов или выполнения workflow.

## Контроль стоимости LLM

ASMO реализует несколько защитных механизмов для предотвращения чрезмерного использования LLM и расходов.

### Стратегия Session-First

Session-режим (Claude CLI) -- провайдер по умолчанию с $0 стоимостью. API-режим используется только как fallback или при явном указании через `--use-api`.

### Маршрутизация моделей по сложности

Задачи направляются к соответствующему уровню модели на основе оценки сложности, обеспечивая резервирование дорогих моделей для задач, которые в них нуждаются.

| Сложность | Модель | Обоснование |
|-----------|--------|-------------|
| 0--30 | Haiku | Тривиальные/простые задачи -- быстрые, дешёвые ответы |
| 31--70 | Sonnet | Сбалансированное качество для средних/сложных задач |
| 71--100 | Opus | Полная мощь рассуждения для enterprise-задач |

### Circuit Breaker

Компонент `CircuitBreaker` мониторит последовательные сбои агентов. После 5 последовательных сбоев в пути выполнения circuit breaker срабатывает и останавливает дальнейшие LLM-вызовы для этого пути. Это предотвращает:

- Неконтролируемые расходы на API от повторяющихся неудачных запросов
- Бесконечные циклы повторов при постоянных ошибках
- Напрасные вычисления при невосстановимых сбоях

Circuit breaker сбрасывается после настраиваемого периода охлаждения.

### Настраиваемые лимиты

- **Максимум повторов на шаг агента:** настраивается через `IterationManager`
- **Таймауты:** поrequestные и пофазные таймауты предотвращают зависание операций
- **Стратегия backoff:** экспоненциальный backoff между повторами снижает нагрузку на провайдеров

## Работа с данными

### Без постоянного хранения кода

ASMO не сохраняет пользовательский исходный код на постоянной основе. Код обрабатывается в памяти при выполнении workflow и удаляется при его завершении. Постоянно хранятся только метаданные (метрики, статус задач, оценки сложности).

### Только локальные метрики

Метрики выполнения хранятся в локальной базе SQLite, управляемой `MetricsPersister`. Эти данные никогда не покидают машину пользователя.

### Без телеметрии

ASMO не собирает и не передаёт никакую телеметрию, аналитику использования или отчёты об ошибках. Все данные остаются локальными.

### Управление API-ключами

- API-ключи считываются исключительно из переменных окружения (`ANTHROPIC_API_KEY`)
- Ключи никогда не записываются в файлы конфигурации, логи или базы метрик
- Файл `.env` включён в `.gitignore` по умолчанию
- Никакие API-ключи не встроены в исходный код или шаблоны пакета

### Санитизация логов

- Чувствительные данные (API-ключи, токены, учётные данные) никогда не включаются в вывод логов
- Verbose-режим (`--verbose`) выводит операционные детали, но не секретные значения
- Сообщения об ошибках санитизируются перед отображением

## Безопасность MCP

### Ролевой доступ

Определение роли каждого агента указывает, к каким MCP-серверам он может обращаться. Агент не может вызвать MCP-сервер, не указанный в списке разрешённых серверов его роли.

```
Agent (role: developer) -> разрешены: [filesystem, context7, github]
Agent (role: tester)    -> разрешены: [filesystem, playwright]
Agent (role: devops)    -> разрешены: [filesystem, github, render, vercel]
```

### Активация по приоритету

MCP-серверы активируются на основе уровня приоритета:

- **P0-серверы** (memory, supabase, filesystem) всегда загружены
- **P1-серверы** (context7, github, playwright) активируются по требованию, когда фаза workflow их запрашивает
- **P2-серверы** (render, vercel) активируются только для workflow деплоя

### Graceful degradation

При недоступности MCP-сервера ASMO деградирует мягко:

- Агент логирует предупреждение и продолжает работу без недоступного инструмента
- Выполнение workflow не прерывается при некритичных сбоях MCP
- P0-сбои могут вызвать провал workflow, если отсутствующая возможность критична
