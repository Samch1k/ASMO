# Карта интеграций

## MCP-серверы

ASMO интегрируется с 8 MCP-серверами (Model Context Protocol), организованными по приоритету активации. Агенты обращаются к MCP-серверам через метод `requestMCP()` класса `BaseAgent`.

| Приоритет | Сервер | Назначение | Основные потребители |
|-----------|--------|------------|---------------------|
| P0 | memory | Хранилище графа знаний | Все агенты |
| P0 | supabase | Интеграция с базой данных | Data Architect, Developer |
| P0 | filesystem | Файловые I/O-операции | Все агенты |
| P1 | context7 | Поиск документации библиотек | Developer, Architect |
| P1 | github | Управление репозиторием | DevOps, Code Reviewer |
| P1 | playwright | Автоматизация браузера (E2E-тесты) | Tester, UI Developer |
| P2 | render | Деплой бэкенда и логи | DevOps |
| P2 | vercel | Деплой фронтенда и логи | DevOps |

**Уровни приоритета:**

- **P0 -- Всегда активны.** Загружаются при старте. Необходимы для основной функциональности.
- **P1 -- По требованию.** Активируются, когда фаза workflow их запрашивает.
- **P2 -- Опциональные.** Активируются только для workflow, связанных с деплоем. Graceful degradation при недоступности.

### Паттерн доступа к MCP

```typescript
// Внутри реализации агента
const files = await this.requestMCP('filesystem', 'readDirectory', { path: './src' })
const docs = await this.requestMCP('context7', 'getLibraryDocs', { library: 'react' })
```

Ролевые ограничения гарантируют, что агенты могут обращаться только к MCP-серверам, разрешённым в определении их роли.

## LLM-провайдеры

ASMO поддерживает три режима LLM-провайдера с автоматическим fallback.

| Режим | Провайдер | Стоимость | Активация |
|-------|-----------|-----------|-----------|
| Session | Claude CLI (`claude -p`) | $0 | По умолчанию (требует аутентифицированный CLI) |
| API | Anthropic SDK (`anthropic`) | Плата за токен | Флаг `--use-api` или Session недоступен |
| Heuristics | Движок keyword-анализа | $0 | Флаг `--no-llm` или оба провайдера недоступны |

### Цепочка fallback

```
Session ($0) ──сбой──> API (плата за токен) ──сбой──> Heuristics ($0, ~65% точности)
```

Session-режим всегда пробуется первым, если не переопределён явно. Если Claude CLI не установлен или не аутентифицирован, ASMO переключается на API-провайдер (требует `ANTHROPIC_API_KEY` в окружении). Если оба недоступны, keyword-эвристики обеспечивают оффлайн-функциональность со сниженной точностью.

### Маршрутизация моделей по сложности

LLM-провайдер направляет запросы к разным моделям Claude на основе оценки сложности задачи.

| Оценка сложности | Модель | Применение |
|------------------|--------|------------|
| 0--30 (trivial/simple) | Haiku | Быстрые, дешёвые операции |
| 31--70 (medium/complex) | Sonnet | Баланс качества и стоимости |
| 71--100 (complex/enterprise) | Opus | Максимальные возможности рассуждения |

Маршрутизация применяется к обоим режимам -- Session и API. Выбор модели автоматический и основан на `ComplexityScore`, полученном при анализе задачи.

## Интеграция с Claude Code

ASMO интегрируется с Claude Code через три механизма:

### CLAUDE.md

Файл `CLAUDE.md` проекта предоставляет Claude Code:

- Обзор системы и количество ресурсов
- Доступные команды и workflow
- Когда использовать ASMO, а когда работать напрямую
- Детали конфигурации окружения

### Интеграция через хуки

ASMO регистрирует хук, запускающий `asmo suggest` при обработке задачи Claude Code. Вывод хука рекомендует, использовать ли оркестрацию ASMO или обработать задачу напрямую.

```bash
# Вызов хука (автоматический)
asmo suggest "<обнаруженная задача>"
# Возвращает JSON: { useAsmo: true/false, confidence, workflow, ... }
```

### Директория конфигурации

ASMO читает конфигурацию проектного уровня из `.cursor/config`, если она доступна, обеспечивая по-проектную кастомизацию workflow, назначений агентов и маппингов навыков.

## Цепочка fallback конфигурации

Значения конфигурации разрешаются через три уровня, где более ранние уровни имеют приоритет.

```
.cursor/config (проект)
    │
    ▼ fallback
~/.asmo/config (пользователь)
    │
    ▼ fallback
встроенные шаблоны (шаблоны пакета)
```

### Источники конфигурации

| Источник | Путь | Область | Содержимое |
|----------|------|---------|------------|
| Проект | `.cursor/config/` | По проекту | Переопределения workflow, конфигурация агентов, маппинги навыков |
| Пользователь | `~/.asmo/config/` | По пользователю | Предпочтения по умолчанию, ссылка на API-ключи, кастомные роли |
| Встроенные | `packages/core/templates/` | Системный | 27 шаблонов workflow, 25 определений ролей, 55 файлов навыков |

## Интеграция сборки и разработки

| Инструмент | Назначение | Конфигурация |
|-----------|------------|-------------|
| pnpm | Управление workspace и зависимостями | `pnpm-workspace.yaml` |
| turbo | Параллельное выполнение задач (build, test, lint) | `turbo.json` |
| tsup | Бандлинг TypeScript | `tsup.config.ts` по пакету |
| Jest | Unit- и интеграционное тестирование | `jest.config.ts`, `tests/setup.ts` |

### Команды сборки

```bash
pnpm build        # Сборка всех пакетов (turbo)
pnpm test         # Запуск всех тестов (jest)
pnpm lint         # Линтинг всех пакетов
pnpm typecheck    # Проверка типов TypeScript
```
