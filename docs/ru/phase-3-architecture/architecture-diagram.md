# Архитектурная схема

## Диаграмма 1: Логические блоки

```
┌─────────────────────────────────────────────────┐
│               Слой CLI (packages/cli)             │
│   asmo run ─ suggest ─ analyze ─ workflow         │
└──────────────────────┬──────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────┐
│       Слой оркестрации (packages/core)            │
│                                                    │
│  A. Анализ задач и маршрутизация                   │
│     ComplexityAnalyzer -> WorkflowSelector          │
│     -> TaskRouter -> SkillMatcher                   │
│     -> AgentRegistry (25 агентов)                   │
│                                                    │
│  B. Движок выполнения Workflow                      │
│     WorkflowEngine <- DynamicOrchestrator           │
│     PhaseManager (6 фаз, настраиваемо)                          │
│     IterationManager (повтор + backoff)            │
│     ApprovalCheckpoint (YOLO-режим)                │
│     AgentExecutor + CircuitBreaker                 │
│     PrincipleValidators (Bob/Winston/John)         │
│     ContextCascade (межфазный поток данных)        │
│                                                    │
│  C. Мультиагентное взаимодействие                   │
│     Sequential - PartySession - Brainstorm          │
│     AdversarialReview                              │
│                                                    │
│  D. Адаптивная детекция фаз                        │
│     ContextAnalyzer -> PhaseDetector (LLM)          │
│                                                    │
│  E. Метрики и цикл обучения                        │
│     MetricsCollector -> MetricsPersister             │
│     -> MetricsOptimizer -> LearningLoop             │
│                                                    │
│  F. Конфигурация и шаблоны                         │
│     ConfigManager (3-уровневый) -> ConfigLoader    │
│     RoleManager - SkillMDLoader - PromptLoader     │
└──────────────────────┬──────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────┐
│             Слой агентов (25 агентов)              │
│   BaseAgent: callLLM(), callLLMForJSON(),          │
│              createResult(), createArtifact(),      │
│              requestMCP(), setRole()                │
└──────────┬───────────────────────┬──────────────┘
           │                       │
┌──────────▼──────────┐  ┌────────▼──────────────────┐
│   LLM-провайдер      │  │       MCP Bridge            │
│   Session ($0)       │  │  memory - supabase - fs     │
│   API ($$)           │  │  context7 - github          │
│   Heuristics         │  │  playwright - render - vercel│
└─────────────────────┘  └──────────────────────────────┘
```

## Сводная таблица блоков

| Блок | Ключевые файлы | Ответственность |
|------|----------------|-----------------|
| **A. Анализ задач** | `ComplexityAnalyzer.ts`, `WorkflowSelector.ts`, `TaskRouter.ts`, `SkillMatcher.ts`, `AgentRegistry.ts` | Оценка сложности задачи (0--100), выбор workflow, подбор навыков, назначение агентов |
| **B. Движок Workflow** | `WorkflowEngine.ts`, `DynamicOrchestrator.ts`, `PhaseManager.ts`, `IterationManager.ts`, `ApprovalCheckpoint.ts`, `AgentExecutor.ts`, `CircuitBreaker.ts`, `ContextCascade.ts` | Выполнение многофазных workflow с повторами, гейтами одобрения, circuit breaking и межфазным потоком данных |
| **C. Коллаборация** | `SequentialCollaboration.ts`, `PartySession.ts`, `BrainstormCollaboration.ts`, `AdversarialReview.ts` | Паттерны мультиагентного взаимодействия внутри одной фазы |
| **D. Детекция фаз** | `ContextAnalyzer.ts`, `PhaseDetector.ts` | Определение оптимальной точки входа при подключении к workflow в середине |
| **E. Метрики** | `MetricsCollector.ts`, `MetricsPersister.ts`, `MetricsOptimizer.ts`, `LearningLoop.ts` | Сбор данных выполнения, сохранение в SQLite, передача в оптимизатор |
| **F. Конфигурация** | `ConfigManager.ts`, `ConfigLoader.ts`, `RoleManager.ts`, `SkillMDLoader.ts`, `PromptLoader.ts` | Загрузка и объединение 3-уровневой конфигурации: проект, пользователь, встроенные |

## 10 ключевых архитектурных решений

### ADR-001: Двойная LLM-стратегия

**Решение:** Использовать Session-режим (Claude CLI, `claude -p`) как основной LLM-провайдер с $0 стоимостью, с Anthropic API в качестве платного fallback.

**Обоснование:** Разработчики, использующие Claude Code, уже имеют аутентифицированную CLI-сессию. Её использование устраняет расходы на API для большинства операций. API-fallback обеспечивает доступность при отсутствии CLI.

**Цепочка fallback:** Session ($0) -> API (плата за токен) -> Heuristics ($0, ~65% точности)

### ADR-002: Нативная TypeScript-оркестрация

**Решение:** Заменить LangGraph собственным `DynamicOrchestrator` на TypeScript.

**Обоснование:** LangGraph вносил тяжёлую зависимость со своей моделью выполнения. Нативный оркестратор даёт полный контроль над управлением фазами, логикой повторов и потоком контекста без привязки к фреймворку.

### ADR-003: LLM-First интеллект

**Решение:** Использовать LLM-вывод как основной механизм для анализа сложности, детекции фаз, выбора workflow и подбора навыков.

**Обоснование:** LLM-анализ достигает примерно 85% точности по сравнению с примерно 65% для keyword-эвристик. Эвристический путь остаётся как оффлайн-fallback.

### ADR-004: 3-уровневый каскад конфигурации

**Решение:** Конфигурация разрешается через три слоя: проект (`.cursor/config`), пользователь (`~/.asmo/config`), встроенные шаблоны.

**Обоснование:** Позволяет кастомизацию по проектам с сохранением разумных значений по умолчанию. Команды могут переопределять шаблоны workflow, определения ролей и конфигурации агентов на любом уровне.

### ADR-005: Singleton-паттерн

**Решение:** Использовать singleton-фабрики (`getX()` / `resetX()`) для 17+ stateful-сервисов.

**Обоснование:** Сервисы типа LLM-провайдера, сборщика метрик и менеджера конфигурации требуют единственных экземпляров. Функция `resetX()` обеспечивает изоляцию тестов без сложной инъекции зависимостей.

### ADR-006: YOLO-режим

**Решение:** Автоматически обходить чекпоинты одобрения при оценке сложности задачи ниже 30.

**Обоснование:** Тривиальные задачи (опечатки, простые переименования) не нуждаются в подтверждении человеком. Порог настраивается и может быть полностью отключён.

### ADR-007: Circuit Breaker

**Решение:** После 5 последовательных сбоев агента circuit breaker срабатывает и останавливает дальнейшие LLM-вызовы для данного пути выполнения.

**Обоснование:** Предотвращает неконтролируемые расходы на API и бесконечные циклы повторов при постоянных сбоях агента или LLM-провайдера.

### ADR-008: Context Cascade

**Решение:** Результаты фаз передаются вперёд через структурированный объект `ContextCascade`, чтобы каждая фаза имела доступ к выходам всех предыдущих фаз.

**Обоснование:** Многофазные workflow требуют общего контекста. Явный поток данных исключает скрытое состояние и упрощает отладку.

### ADR-009: Интеграция MCP

**Решение:** Интегрировать 8 MCP-серверов, организованных по приоритету: P0 (всегда активны), P1 (по требованию), P2 (опциональные).

**Обоснование:** MCP предоставляет стандартизированный протокол доступа к инструментам. Приоритетная активация избегает загрузки ненужных серверов и обеспечивает graceful degradation.

### ADR-010: Principle Validators

**Решение:** Три именованных валидатора (Bob, Winston, John) действуют как quality gates при выполнении workflow.

**Обоснование:** Автоматическая принципиальная валидация отлавливает типичные проблемы (стиль кода, паттерны безопасности, анти-паттерны производительности) до завершения фазы, сокращая переделки.
