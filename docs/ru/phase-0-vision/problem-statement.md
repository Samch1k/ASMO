# Постановка проблемы

## Контекст

Разработка с помощью ИИ стала мейнстримом, однако инструментарий остается
фундаментально однопоточным: один человек, один помощник, один промпт за раз.
По мере роста сложности задач -- затрагивающих множество файлов, аспектов и
доменов -- эта модель перестает работать. Разработчики прибегают к произвольным
цепочкам промптов, ручной координации агентов и дорогостоящим API-вызовам
без структурированных контрольных точек качества.

## Проблемы, которые решает ASMO

### 1. Высокая стоимость LLM API-вызовов

Каждое взаимодействие с облачным LLM стоит токенов. При итеративной разработке --
отладке, рефакторинге, тестировании -- эти затраты быстро накапливаются.
Большинство фреймворков не предлагают маршрутизацию с учетом стоимости.

### 2. Отсутствие структурированных воркфлоу для сложных задач

Когда задача требует архитектурного ревью, реализации, тестирования и
документации, разработчик должен вручную оркестрировать каждый шаг. Нет
стандартного способа определить, переиспользовать или адаптировать
многошаговые AI-воркфлоу.

### 3. Зависимость от внешних фреймворков

Существующие фреймворки оркестрации (LangGraph, CrewAI, AutoGen) требуют
Python, вводят тяжелые деревья зависимостей и навязывают собственные
модели выполнения. TypeScript-проекты сталкиваются с языковым барьером.

### 4. Ручная маршрутизация между специализированными агентами

Решение о том, какого агента или промпт использовать для какой подзадачи,
целиком лежит на разработчике. Нет автоматического анализа сложности или
сопоставления агента с задачей.

### 5. Отсутствие адаптивного анализа сложности

Все задачи обрабатываются одинаково, независимо от фактической сложности.
Однострочное исправление опечатки получает тот же тяжеловесный конвейер,
что и кроссервисный рефакторинг.

### 6. Отсутствие контрольных точек качества

Без структурированных чекпоинтов AI-сгенерированный код обходит ревью,
тестирование и архитектурные стандарты. Ошибки распространяются без
контроля через цикл разработки.

## Маппинг проблем на решения

| Проблема | Решение ASMO | Компонент |
|----------|-------------|-----------|
| Высокая стоимость LLM API | Dual LLM: Session ($0) -> API -> Heuristics | `LLMProviderFactory` |
| Нет структурированных воркфлоу | 27 преднастроенных воркфлоу с управлением фазами | `WorkflowEngine`, `PhaseManager` |
| Зависимость от внешних фреймворков | Native TypeScript, без Python, минимальные зависимости | Ядро архитектуры |
| Ручная маршрутизация агентов | Автоматическая оценка и маршрутизация (шкала 0--100) | `ComplexityAnalyzer`, `TaskRouter` |
| Нет анализа сложности | 5-уровневая классификация с анализом ключевых слов + LLM | `ComplexityAnalyzer` |
| Нет контрольных точек качества | Чекпоинты одобрения, adversarial review, circuit breaker | `ApprovalCheckpoint`, `CircuitBreaker` |
| Повторная работа при частичных задачах | Адаптивная детекция фаз по существующим артефактам | `PhaseDetector`, `ContextAnalyzer` |
| Нет видимости затрат | Сбор метрик и отслеживание стоимости по воркфлоу | `MetricsCollector` |

## Кого это затрагивает

- **Индивидуальные разработчики**, которые платят за API из собственного кармана.
- **Инженерные команды**, у которых нет повторяемых AI-воркфлоу.
- **Архитекторы**, которым нужны структурированные многосторонние процессы проектирования.
- **DevOps-инженеры**, которые хотят автоматизированные, проверяемые конвейеры деплоя.

## Как выглядит успех

Разработчик выполняет одну команду -- `asmo run "реализовать OAuth2 с PKCE"` --
и система автоматически анализирует сложность, выбирает подходящий воркфлоу,
направляет работу через нужных агентов, обеспечивает контрольные точки качества
и выдает протестированный, проверенный результат. Итоговая стоимость API: $0
в Session-режиме.
