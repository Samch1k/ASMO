# Метрики успеха

Метрики организованы по четырем категориям, каждая с горизонтом 6 месяцев от
первого публичного релиза.

## 1. Принятие продукта (Adoption)

| Метрика | Цель (6 месяцев) | Измерение |
|---------|------------------|-----------|
| npm-загрузки | 1 000 / месяц | Статистика npm-реестра |
| GitHub stars | 500 | Репозиторий GitHub |
| Активные пользователи | 100 | CLI-телеметрия (opt-in) |
| Вклады сообщества | 20 вмерженных PR | Pull request-ы на GitHub |

**Обоснование:** Эти пороги представляют здоровый проект с открытым кодом на
ранней стадии с органическим ростом. Цель 1 000 загрузок/месяц учитывает как
прямые установки, так и использование в CI/CD-конвейерах.

## 2. Производительность (Performance)

| Метрика | Цель | Базовая линия |
|---------|------|--------------|
| Время выполнения воркфлоу | -20% по сравнению с ручной оркестрацией | Измерение против эквивалентных ручных многошаговых промптов |
| Снижение затрат на LLM | -30% через Session-провайдер | По сравнению с чистым API-режимом для тех же задач |
| Задержка анализа задач | < 5 секунд | ComplexityAnalyzer end-to-end |
| Задержка выбора воркфлоу | < 2 секунды | WorkflowSelector end-to-end |

**Обоснование:** Session-провайдер ($0) -- основной рычаг снижения затрат.
Улучшение времени выполнения достигается за счет устранения ручной координации
и включения параллельного выполнения агентов.

## 3. Качество (Quality)

| Метрика | Цель | Измерение |
|---------|------|-----------|
| Частота баг-репортов | < 5 / неделю | Issues на GitHub с меткой `bug` |
| Тестовое покрытие | >= 80% | Jest coverage report |
| Удовлетворенность пользователей | >= 4.5 / 5 | Опрос после воркфлоу (opt-in) |
| Процент успеха воркфлоу | >= 90% | Завершение без срабатывания CircuitBreaker |
| Точность классификации | >= 77% | ComplexityAnalyzer против человеческих меток |

**Обоснование:** Целевая точность классификации 77% отражает текущие возможности
LLM-анализа. Целевое покрытие 80% согласовано с NFR по поддерживаемости.

## 4. Опыт разработчика (Developer Experience)

| Метрика | Цель | Измерение |
|---------|------|-----------|
| Полнота документации | 100% | Все публичные API задокументированы |
| Время до первого воркфлоу | < 5 минут | От `npm install` до первого успешного `asmo run` |
| Время решения issues | < 48 часов | Медиана от открытия issue до вмерженного фикса |
| Ясность сообщений об ошибках CLI | >= 90% содержат действие | Обратная связь пользователей по сообщениям об ошибках |

**Обоснование:** Метрики опыта разработчика гарантируют, что ASMO остается
доступным. Цель 5-минутного онбординга означает, что разработчик может
установить, аутентифицироваться и запустить первый воркфлоу без изучения
обширной документации.

## Отслеживание и ревью

- Метрики пересматриваются ежемесячно в течение первых 6 месяцев после релиза.
- Дашборд: GitHub Insights + npm-статистика + данные MetricsCollector.
- Корректировка курса запускается, если любая метрика падает ниже 70% от цели
  в течение двух последовательных месяцев.
