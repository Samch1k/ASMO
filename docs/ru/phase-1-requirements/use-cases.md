# Сценарии использования

Пять основных сценариев использования демонстрируют возможности ASMO на
разных уровнях сложности и с разными паттернами коллаборации.

---

## UC-1: Простой баг-фикс (YOLO Mode)

**Сложность:** Trivial (оценка ~15)
**Воркфлоу:** `bug_fix_workflow`
**Агенты:** Debugger, Developer
**Целевое время:** < 60 секунд

### Сценарий

Разработчик обнаруживает, что утилитарная функция возвращает `null` вместо
пустого массива, когда результатов нет. Это вызывает `TypeError: Cannot read
properties of null` в нижестоящем коде.

### Поток

1. Разработчик запускает: `asmo run "Исправить null-возврат в getUserRoles() -- должен возвращать пустой массив"`
2. `ComplexityAnalyzer` оценивает задачу в 15 баллов (trivial: одна функция, понятное исправление).
3. `WorkflowSelector` выбирает `bug_fix_workflow`.
4. `ApprovalCheckpoint` активирует **YOLO-режим** (оценка < 30) -- автоматическое одобрение, без промпта.
5. Агент debugger находит `getUserRoles()`, подтверждает путь с null-возвратом.
6. Агент developer применяет исправление: `return results ?? []`.
7. Результат возвращается пользователю менее чем за 60 секунд.

### Ключевое поведение

- Одобрение человека не требуется (YOLO-обход).
- Минимальное участие агентов (2 агента, последовательно).
- CircuitBreaker и повторы доступны, но вряд ли сработают.

## UC-2: Реализация фичи с брейнштормом

**Сложность:** Complex (оценка ~75)
**Воркфлоу:** `feature_implementation_full`
**Агенты:** Architect, Developer, Tester
**Оценка времени:** 15--30 минут

### Сценарий

Команде нужно реализовать систему уведомлений, поддерживающую email, SMS и
push-уведомления с подключаемой архитектурой провайдеров.

### Поток

1. Разработчик запускает: `asmo run "Реализовать многоканальную систему уведомлений с подключаемыми провайдерами"`
2. `ComplexityAnalyzer` оценивает в 75 баллов (complex: множество аспектов, архитектурные решения, новая подсистема).
3. `WorkflowSelector` выбирает `feature_implementation_full`.
4. `ApprovalCheckpoint` запрашивает подтверждение пользователя (оценка >= 30).
5. **Фаза дизайна:** Агент architect создает:
   - Определения интерфейсов для `NotificationProvider`, `NotificationChannel`
   - ADR, документирующий паттерн подключаемых провайдеров vs. альтернативы
   - Граф зависимостей и точки интеграции
6. **Фаза реализации:** Агент developer пишет:
   - Ядро сервиса уведомлений
   - Реализации провайдеров email, SMS, push
   - Конфигурацию и фабричный паттерн для регистрации провайдеров
7. **Фаза тестирования:** Агент tester генерирует:
   - Юнит-тесты для каждого провайдера
   - Интеграционные тесты для конвейера уведомлений
   - Тесты граничных случаев (отказ провайдера, повтор, откат)
8. `ContextCascade` обеспечивает, что каждая фаза строится на результатах предыдущей.

### Ключевое поведение

- ADR (Architecture Decision Record) генерируется автоматически.
- Все три агента вносят вклад последовательно, с передачей контекста между фазами.
- Целевое тестовое покрытие >= 80% для нового кода.

## UC-3: Adversarial код-ревью

**Сложность:** Medium (оценка ~50)
**Воркфлоу:** `adversarial_review_workflow`
**Агенты:** Code Reviewer, Developer
**Оценка времени:** 10--15 минут

### Сценарий

Критический модуль аутентификации был обновлен. Перед мержем команда хочет
провести adversarial review, который активно ищет уязвимости.

### Поток

1. Разработчик запускает: `asmo run "Adversarial review изменений модуля auth в src/auth/"`
2. `ComplexityAnalyzer` оценивает в 50 баллов (medium: ограниченный скоуп, но чувствительный к безопасности).
3. `WorkflowSelector` выбирает `adversarial_review_workflow`.
4. `ApprovalCheckpoint` запрашивает подтверждение пользователя.
5. **Фаза ревью (Уровень 1 -- Стандартный):** Агент code-reviewer проверяет:
   - Корректность логики
   - Полноту обработки ошибок
   - Покрытие валидации ввода
6. **Фаза ревью (Уровень 2 -- Adversarial):** Агент code-reviewer пытается:
   - Векторы атак инъекцией
   - Сценарии обхода аутентификации
   - Эксплуатацию race condition
   - Манипуляцию токенами
7. **Фаза ревью (Уровень 3 -- Эскалированный):** Если минимальное количество проблем не найдено, ревьюер эскалирует:
   - Анализ более широкого контекста (что вызывающий код предполагает об этом модуле)
   - Уязвимости в цепочке зависимостей
   - Поверхности атак на основе конфигурации
8. Агент developer формирует рекомендации по исправлению для каждой найденной проблемы.

### Ключевое поведение

- **Минимум проблем:** Ревьюер обязан найти не менее N проблем (настраивается).
- **3-уровневая эскалация:** Каждый уровень расширяет исследуемую поверхность атаки.
- Проблемы категоризируются по серьезности: critical, major, minor.

## UC-4: TEA-планирование тестов

**Сложность:** Complex (оценка ~70)
**Воркфлоу:** `tea_planning_workflow`
**Агенты:** Test Planner, Risk Analyst, Test Designer
**Оценка времени:** 3--5 часов

### Сценарий

Новый модуль обработки платежей нуждается в комплексной стратегии тестирования
до начала разработки. Модуль обрабатывает кредитные карты, ACH-переводы и
криптовалютные платежи.

### Поток

1. QA-лид запускает: `asmo run "Создать тест-план для модуля обработки платежей с картами, ACH и крипто"`
2. `ComplexityAnalyzer` оценивает в 70 баллов (complex: множество способов оплаты, комплаенс, безопасность).
3. `WorkflowSelector` выбирает `tea_planning_workflow`.
4. **Фаза оценки рисков:**
   - Идентификация зон риска: PCI-комплаенс, атомарность транзакций, конвертация валют
   - Приоритизация по влиянию и вероятности
   - Маппинг рисков на требования к тестовому покрытию
5. **Фаза стратегии:**
   - Определение уровней тестирования: unit, интеграционные, E2E, нагрузочные, безопасности
   - Распределение усилий по способам оплаты
   - Определение требований к окружению (sandbox-аккаунты, тестовые карты)
6. **Фаза дизайна тестов:**
   - Создание спецификаций тест-кейсов для каждого способа оплаты
   - Определение требований к данным и фикстур
   - Создание матрицы прослеживаемости: требование -> тест-кейс -> риск

### Ключевое поведение

- Создает полный документ стратегии тестирования.
- Оценка рисков управляет приоритизацией тестов.
- Результат структурирован для непосредственного использования QA-командами.

## UC-5: Архитектурный дизайн с Party Mode

**Сложность:** Enterprise (оценка ~90)
**Воркфлоу:** `architecture_design`
**Агенты:** Несколько агентов с ролью architect (Party Mode)
**Оценка времени:** 30--60 минут

### Сценарий

Инженерная команда мигрирует с монолитного приложения на микросервисы. Им нужна
целевая архитектура, которая адресует границы сервисов, владение данными,
паттерны коммуникации и стратегию деплоя.

### Поток

1. Архитектор запускает: `asmo run "Спроектировать микросервисную архитектуру для миграции e-commerce платформы"`
2. `ComplexityAnalyzer` оценивает в 90 баллов (enterprise: влияние на всю систему, множество доменов).
3. `WorkflowSelector` выбирает `architecture_design`.
4. **Фаза брейншторма (Party Mode):** Несколько агентов вносят вклад одновременно:
   - **Агент A** предлагает границы сервисов на основе доменов
   - **Агент B** выступает за событийную коммуникацию (Kafka/NATS)
   - **Агент C** аргументирует за синхронные API с circuit breaker
   - **Агент D** оценивает владение данными и модели консистентности
5. **Фаза конвергенции:** Агенты синтезируют перспективы в:
   - Определения границ сервисов с ограниченными контекстами
   - Выбор паттерна коммуникации с анализом компромиссов
   - Стратегию миграции данных (паттерн strangler fig)
   - ADR для каждого крупного решения
6. **Фаза валидации:** Adversarial review предложенной архитектуры:
   - Анализ единых точек отказа
   - Идентификация узких мест масштабируемости
   - Оценка операционной сложности

### Ключевое поведение

- **Party Mode:** Агенты работают параллельно, создавая независимые перспективы перед конвергенцией.
- Генерируется несколько ADR, по одному на каждое крупное архитектурное решение.
- Adversarial-валидация гарантирует, что дизайн подвергается проверке перед принятием.
