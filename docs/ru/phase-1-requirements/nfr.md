# Нефункциональные требования (НФТ)

Этот документ определяет нефункциональные требования для системы ASMO,
охватывая производительность, надежность, масштабируемость, поддерживаемость
и безопасность.

---

## NFR-1: Производительность

| ID | Требование | Целевое значение |
|----|-----------|-----------------|
| NFR-1.1 | Задержка анализа задачи (ComplexityAnalyzer) | < 5 секунд end-to-end |
| NFR-1.2 | Задержка выбора воркфлоу (WorkflowSelector) | < 2 секунды |
| NFR-1.3 | Инициализация WorkflowEngine | < 1 секунда |
| NFR-1.4 | Время первого запуска CLI (cold start) | < 3 секунды |
| NFR-1.5 | Потребление памяти при выполнении воркфлоу | < 512 MB RSS |
| NFR-1.6 | Heuristics-режим (без LLM) | < 100 мс для анализа задачи |

**Обоснование:** Производительность критична для CLI-инструмента. Задержки
свыше 5 секунд на анализ нарушают рабочий поток разработчика. Heuristics-режим
должен быть мгновенным, так как он используется для офлайн-работы.

---

## NFR-2: Надежность

| ID | Требование | Целевое значение |
|----|-----------|-----------------|
| NFR-2.1 | Процент успеха воркфлоу (без внешних ошибок) | >= 95% |
| NFR-2.2 | Graceful degradation при отказе LLM-провайдера | Автоматический откат в течение 5 секунд |
| NFR-2.3 | CircuitBreaker предотвращает каскадные отказы | Срабатывание после 5 последовательных ошибок |
| NFR-2.4 | IterationManager обеспечивает устойчивость к транзиентным ошибкам | До 3 повторов на фазу |
| NFR-2.5 | Обработка ошибок возвращает действенные сообщения | 100% ошибок категоризированы (transient/permanent/config) |
| NFR-2.6 | Потеря данных при прерывании выполнения | 0 -- артефакты завершенных фаз сохраняются |

**Обоснование:** Надежность критична, поскольку ASMO оркестрирует длительные
процессы. Отказ на поздней фазе не должен уничтожать результаты ранних фаз.
Автоматический откат LLM-провайдера обеспечивает непрерывность работы.

---

## NFR-3: Масштабируемость

| ID | Требование | Целевое значение |
|----|-----------|-----------------|
| NFR-3.1 | Количество одновременных агентов (Party Mode) | До 6 агентов параллельно |
| NFR-3.2 | Количество фаз в одном воркфлоу | До 10 фаз |
| NFR-3.3 | Размер контекста, передаваемого через ContextCascade | До 100 KB на фазу |
| NFR-3.4 | Количество воркфлоу в каталоге | До 100 (текущее: 27) |
| NFR-3.5 | Количество навыков в каталоге | До 200 (текущее: 55) |

**Обоснование:** Текущие значения (27 воркфлоу, 55 навыков) далеки от лимитов.
Архитектура должна поддерживать рост каталога без деградации производительности.
Party Mode ограничен 6 агентами для предотвращения чрезмерного потребления
ресурсов LLM.

---

## NFR-4: Поддерживаемость

| ID | Требование | Целевое значение |
|----|-----------|-----------------|
| NFR-4.1 | Тестовое покрытие кода | >= 80% (line coverage) |
| NFR-4.2 | Добавление нового агента | Не требует изменения ядра (plugin-паттерн) |
| NFR-4.3 | Добавление нового воркфлоу | Только JSON-шаблон + регистрация |
| NFR-4.4 | Добавление нового навыка | Только файл SKILL.md + запись в зависимостях |
| NFR-4.5 | Документация публичного API | 100% документировано |
| NFR-4.6 | Линтинг и форматирование | ESLint + Prettier, 0 ошибок в CI |
| NFR-4.7 | TypeScript strict mode | Включен, 0 any-типов в новом коде |

**Обоснование:** Расширяемость -- ключевое свойство ASMO. Добавление новых
агентов, воркфлоу и навыков должно быть простой операцией, не требующей
глубоких знаний внутренней архитектуры. Strict TypeScript предотвращает
регрессии типов.

---

## NFR-5: Безопасность

| ID | Требование | Целевое значение |
|----|-----------|-----------------|
| NFR-5.1 | Валидация ввода | 10 проверок, включая инъекции, управляющие символы, null-байты |
| NFR-5.2 | Хранение API-ключей | Только через переменные окружения или `.env` (никогда в коде) |
| NFR-5.3 | Санитизация вывода агентов | Предотвращение инъекций в промпты между фазами |
| NFR-5.4 | MCP-доступ на основе ролей | Каждый агент видит только разрешенные MCP-серверы |
| NFR-5.5 | Логирование чувствительных данных | API-ключи и токены не попадают в логи |
| NFR-5.6 | Зависимости | 0 known critical vulnerabilities в прямых зависимостях |

**Обоснование:** ASMO обрабатывает код и может взаимодействовать с внешними
сервисами через MCP. Инъекции в промпты между фазами -- реальный вектор атаки.
Валидация ввода и ролевой доступ к MCP минимизируют поверхность атаки.
