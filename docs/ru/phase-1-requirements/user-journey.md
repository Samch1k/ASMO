# Путь пользователя

Этот документ описывает end-to-end поток взаимодействия при запуске задачи
через ASMO.

## Диаграмма потока

```
+--------------------+
| Пользователь       |
| имеет задачу       |
+--------+-----------+
         |
         v
+--------+-----------+     +---------------------+
| asmo suggest/run   |---->| Валидация ввода     |
+--------+-----------+     | (10 проверок)       |
         |                 +---------------------+
         v
+--------+-----------+
| ComplexityAnalyzer  |
| Оценка: 0-100      |
| Уровни: 5          |
+--------+-----------+
         |
         v
+--------+-----------+
| WorkflowSelector    |
| 27 воркфлоу         |
+--------+-----------+
         |
         v
+--------+-----------+     +---------------------+
| PhaseDetector       |---->| Сканирование        |
| (опционально)       |     | артефактов          |
+--------+-----------+     | Определение стартовой|
         |                 | фазы                |
         v                 +---------------------+
+--------+-----------+
| ApprovalCheckpoint  |
| YOLO: авто если <30 |
| Иначе: промпт       |
+--------+-----------+
         |
         v
+--------+-----------+
| WorkflowEngine      |
| Фаза 1..N           |
+--------+-----------+
         |
    +----+----+
    |    |    |
    v    v    v
 Agent Agent Agent    <-- AgentExecutor запускает каждого
    |    |    |
    v    v    v
  LLM  LLM  MCP      <-- callLLM() / requestMCP()
    |    |    |
    +----+----+
         |
         v
+--------+-----------+
| ContextCascade      |
| Данные текут между   |
| фазами              |
+--------+-----------+
         |
         v
+--------+-----------+
| MetricsCollector    |
| Запись метрик        |
+--------+-----------+
         |
         v
+--------+-----------+
| Результат            |
| пользователю        |
+---------------------+
```

## Пошаговое описание

### Шаг 1: Ввод задачи

Пользователь предоставляет описание задачи через CLI:

```bash
asmo run "Добавить rate limiting к эндпоинту /api/users с Redis-бэкендом"
```

Альтернативно пользователь может сначала проверить, уместно ли использовать ASMO:

```bash
asmo suggest "Добавить rate limiting к эндпоинту /api/users с Redis-бэкендом"
# Вернет: { "useAsmo": true, "reasoning": "Множество аспектов: API, производительность, кэширование" }
```

### Шаг 2: Валидация ввода

Описание задачи проходит через 10 проверок валидации:

- Проверка на пустоту/null
- Ограничения длины (3--10 000 символов)
- Валидация кодировки UTF-8
- Удаление управляющих символов (безопасность)
- Нормализация пробелов

Некорректный ввод возвращает действенное сообщение об ошибке. Санитизируемые
проблемы (управляющие символы, избыточные пробелы) очищаются автоматически.

### Шаг 3: Анализ сложности

`ComplexityAnalyzer` оценивает задачу, используя двойную стратегию:

1. **LLM-анализ** (основной): Отправляет задачу LLM-провайдеру со
   структурированным промптом. Возвращает оценку, уровень и категорию.
2. **Эвристики по ключевым словам** (откат): Сканирует ключевые слова по
   14 категориям. Применяет модификаторы влияния для high-risk, данных,
   безопасности и производительности.

Результат: Оценка 65, Уровень "complex", Категория "api + performance".

### Шаг 4: Выбор воркфлоу

`WorkflowSelector` сопоставляет результат анализа с 27 определениями воркфлоу.
Выбор учитывает:

- Соответствие категории (api -> `api_design` или `feature_implementation_full`)
- Порог сложности (высокосложные задачи получают мультиагентные воркфлоу)
- Буст ключевых слов (например, "rate limiting" + "Redis" повышают
  `performance_optimization`)

Результат: `feature_implementation_full` (architect + developer + tester).

### Шаг 5: Адаптивная детекция фаз (опционально)

Если проект уже содержит релевантные артефакты, `PhaseDetector` сканирует:

- Существующие архитектурные документы
- Тестовые файлы для целевого модуля
- Git-историю на предмет недавних связанных изменений

Если артефакты дизайна существуют, воркфлоу пропускает фазу архитектуры и
начинает с реализации. Детекция предоставляет оценку уверенности и
альтернативные точки входа.

### Шаг 6: Контрольная точка одобрения

`ApprovalCheckpoint` определяет, нужно ли одобрение человека:

- **YOLO-режим** (оценка < 30): Автоматическое одобрение, без промпта.
  Выполнение начинается немедленно.
- **Стандартный режим** (оценка >= 30): Пользователь видит выбранный воркфлоу,
  агентов и предполагаемые фазы. Подтверждает или отменяет.

```
Воркфлоу: feature_implementation_full
Агенты:  architect, developer, tester
Фазы:   design -> implement -> test -> review
Продолжить? [Y/n]
```

### Шаг 7: Выполнение воркфлоу

`WorkflowEngine` выполняет каждую фазу последовательно (или параллельно для
Party Mode):

1. **Фаза дизайна**: Агент architect создает документ дизайна и определения
   интерфейсов.
2. **Фаза реализации**: Агент developer пишет код по дизайну.
3. **Фаза тестирования**: Агент tester генерирует и запускает тесты.
4. **Фаза ревью**: Агент code-reviewer валидирует результат.

Каждый агент вызывает `callLLM()` для рассуждений и `requestMCP()` для доступа
к инструментам (файловая система, GitHub, браузер).

### Шаг 8: Каскад контекста

`ContextCascade` передает артефакты между фазами:

- Документ дизайна -> Реализация (developer использует интерфейсы из дизайна)
- Результат реализации -> Тестирование (tester пишет тесты для нового кода)
- Все артефакты -> Ревью (reviewer видит полную картину)

### Шаг 9: Обработка ошибок

Если фаза завершается с ошибкой:

- `IterationManager` повторяет до 3 раз.
- `CircuitBreaker` срабатывает после 5 последовательных отказов, останавливая
  воркфлоу.
- Ошибки категоризируются (transient, permanent, configuration) для действенной
  обратной связи.

### Шаг 10: Метрики и вывод

`MetricsCollector` записывает:

- Общее время выполнения
- Длительность каждой фазы
- Использование токенов LLM и стоимость
- Статус успеха/неуспеха

Итоговый результат предоставляется пользователю с резюме того, что было
создано, изменено и протестировано.

## Флаги CLI, изменяющие путь

| Флаг | Влияние на поток |
|------|-----------------|
| `--dry-run` | Останавливается после Шага 4 (только анализ, без выполнения) |
| `--use-api` | Форсирует API-режим на Шаге 3 и Шаге 7 |
| `--no-llm` | Использует только эвристики на Шаге 3 (~65% точность) |
| `--workflow <name>` | Пропускает Шаг 4, использует указанный воркфлоу напрямую |
| `--verbose` | Подробное логирование на каждом шаге |
