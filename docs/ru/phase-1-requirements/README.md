# Фаза 1 -- Обзор требований

Этот документ суммирует восемь функциональных требований (FR), определяющих
возможности ASMO. Каждое FR привязано к конкретным компонентам системы и имеет
связанные критерии приемки, документированные в
[acceptance-criteria.md](./acceptance-criteria.md).

## FR-1: Анализ задач и маршрутизация

**Приоритет: P0**

Система должна анализировать входящие задачи и направлять их в подходящий
воркфлоу и к нужным агентам.

| Компонент | Ответственность |
|-----------|----------------|
| `ComplexityAnalyzer` | Оценивает сложность задачи 0--100 по 5 уровням (trivial, simple, medium, complex, enterprise) |
| `WorkflowSelector` | Выбирает наиболее подходящий воркфлоу из 27 доступных |
| `TaskRouter` | Направляет задачу в выбранный воркфлоу с конфигурацией |
| `SkillMatcher` | Сопоставляет необходимые навыки из каталога в 55 навыков, разрешает зависимости |

Анализ использует двойную стратегию: LLM-классификация (>=77% точности) с
откатом на эвристики по ключевым словам (~65% точности). Система из 14
категорий ключевых слов покрывает trivial, bug_fix, architecture, security,
performance, database, refactoring, api, feature и другие.

## FR-2: Движок выполнения воркфлоу

**Приоритет: P0**

Система должна выполнять многофазные воркфлоу с обработкой ошибок, повторными
попытками и распространением контекста.

| Компонент | Ответственность |
|-----------|----------------|
| `WorkflowEngine` | Ядро оркестрации; фабричный паттерн через `WorkflowEngine.create()` |
| `DynamicOrchestrator` | Адаптирует выполнение на основе условий времени выполнения |
| `PhaseManager` | Управляет переходами между фазами и их порядком |
| `ContextCascade` | Распространяет данные и артефакты между фазами |
| `CircuitBreaker` | Срабатывает после 5 последовательных отказов, предотвращая каскад |
| `IterationManager` | Управляет логикой повторов (максимум 3 повтора на фазу) |

YOLO-режим автоматически одобряет задачи с complexity score < 30 через
`ApprovalCheckpoint`.

## FR-3: Мультиагентная коллаборация

**Приоритет: P0**

Система должна поддерживать множество паттернов коллаборации между 25 агентами.

| Паттерн | Описание | Применение |
|---------|---------|-----------|
| **Sequential** | Агенты выполняются друг за другом, выход передается следующему | Стандартные воркфлоу |
| **PartySession** | Несколько агентов проводят брейншторм одновременно | Архитектурный дизайн |
| **BrainstormingSession** | Структурированная идеация с конвергенцией | Планирование фич |
| **AdversarialReview** | Ревьюер проверяет код с нарастающей строгостью | Качество кода |

Сессии брейншторминга создают артефакты ADR (Architecture Decision Record).
Adversarial review требует обнаружения минимального количества проблем с
3-уровневой эскалацией по серьезности.

## FR-4: Адаптивная детекция фаз

**Приоритет: P1**

Система должна определять подходящую стартовую фазу воркфлоу на основе
существующих артефактов и контекста.

| Компонент | Ответственность |
|-----------|----------------|
| `ContextAnalyzer` | Сканирует существующие файлы, git-историю и состояние проекта |
| `PhaseDetector` | Определяет, какие фазы уже были завершены |

Это предотвращает повторную работу. Если архитектурные документы уже
существуют, система пропускает фазу проектирования и начинает с реализации.
Детекция включает оценку уверенности и альтернативные предложения.

## FR-5: Слой LLM-провайдера

**Приоритет: P0**

Система должна поддерживать три LLM-режима с автоматическим откатом.

| Режим | Стоимость | Требования | Порядок отката |
|-------|-----------|------------|---------------|
| **Session** | $0 | Claude CLI установлен + аутентифицирован | Основной |
| **API** | Pay-per-token | `ANTHROPIC_API_KEY` в `.env` | Вторичный |
| **Heuristics** | $0 | Ничего | Третичный |

`LLMProviderFactory` реализует паттерн синглтона и управляет выбором
провайдера. Session-режим вызывает Claude через подпроцесс `claude -p`.
Heuristics-режим использует только анализ ключевых слов (~65% точности).

## FR-6: MCP-интеграция

**Приоритет: P1**

Система должна интегрироваться с MCP-серверами (Model Context Protocol) для
расширенных возможностей.

Агенты могут вызывать `requestMCP()` для доступа к внешним инструментам.
Поддерживается 8 интеграций с MCP-серверами, включая операции с файловой
системой, GitHub API и автоматизацию браузера. Доступ на основе ролей --
каждый агент видит только релевантные для своей роли MCP-серверы.

## FR-7: Управление конфигурацией

**Приоритет: P1**

Система должна поддерживать трехуровневую иерархию конфигурации.

| Уровень | Источник | Приоритет переопределения |
|---------|---------|--------------------------|
| **Defaults** | Встроенные JSON-шаблоны | Низший |
| **Project** | Каталог `.asmo/` в корне проекта | Средний |
| **Runtime** | Флаги CLI и переменные окружения | Высший |

`ConfigLoader` обрабатывает слияние. Роли определяются в JSON, навыки
в SKILL.md markdown-файлах, воркфлоу в JSON-шаблонах.

## FR-8: Слой персистенции

**Приоритет: P2**

Система должна сохранять состояние воркфлоу, метрики и историю выполнения.

| Хранилище | Назначение |
|-----------|-----------|
| **SQLite** | История выполнения воркфлоу, агрегация метрик |
| **JSON-файлы** | Конфигурация, определения навыков, шаблоны воркфлоу |

`MetricsCollector` записывает данные о производительности для каждого
выполнения воркфлоу. `MetricsOptimizer` использует исторические данные для
улучшения будущего выбора воркфлоу (обучающий цикл подключен к
`WorkflowEngine.initialize()`).
