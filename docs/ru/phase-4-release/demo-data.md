## Задача 1: Исправить опечатку в README

**Ввод:**
```bash
asmo run "Исправить опечатку в README"
```

**Анализ сложности:**
- Оценка: 15
- Уровень: Trivial
- Категории: `trivial`
- Модификаторы влияния: нет

**Выбор workflow:** `bug_fix_workflow`

**Назначенные агенты:** debugger (один агент, минимальный конвейер)

**Примерная длительность:** < 1 минуты

**YOLO-режим:** Да -- оценка ниже порога 30, автоодобрение без чекпоинта. Задача выполняется немедленно без запроса одобрения.

**Ожидаемый результат:** Один файл изменён, тесты не требуются.

---

## Задача 2: Исправить null pointer в обработчике логина

**Ввод:**
```bash
asmo run "Исправить null pointer в обработчике логина"
```

**Анализ сложности:**
- Оценка: 35
- Уровень: Simple
- Категории: `bug_fix`
- Модификаторы влияния: `security_impact` (обработчик логина затрагивает аутентификацию)

**Выбор workflow:** `bug_fix_workflow`

**Назначенные агенты:**
1. **debugger** -- исследует null pointer, определяет корневую причину (отсутствующая проверка на null, обращение к undefined свойству и т.д.)
2. **developer** -- реализует исправление с корректной обработкой null
3. **tester** -- добавляет регрессионный тест для случая null

**Примерная длительность:** 10-15 минут

**YOLO-режим:** Нет -- оценка превышает порог 30, предъявляется чекпоинт одобрения.

**Ожидаемый результат:** 2-3 файла изменены (обработчик + тест), добавлена null-безопасность, тест покрывает конкретный null-сценарий.

## Задача 3: Добавить валидацию email в форму регистрации

**Ввод:**
```bash
asmo run "Добавить валидацию email в форму регистрации"
```

**Анализ сложности:**
- Оценка: 45
- Уровень: Medium
- Категории: `feature`, `api`
- Модификаторы влияния: нет

**Выбор workflow:** `feature_implementation_full`

**Назначенные агенты:**
1. **architect** -- определяет стратегию валидации (клиентская, серверная или обе), выбор regex-паттерна или библиотеки
2. **developer** -- реализует валидацию в компоненте формы и API-эндпоинте
3. **tester** -- создаёт тест-кейсы для валидных email, невалидных форматов, edge-кейсов (Unicode, длинные домены, субадресация)

**Примерная длительность:** 1-2 часа

**YOLO-режим:** Нет -- оценка превышает порог 30.

**Ожидаемый результат:** 3-5 файлов изменены (компонент формы, API-валидатор, тесты), реализована и клиентская, и серверная валидация, комплексные тест-кейсы для edge-кейсов формата email.

## Задача 4: Реализовать OAuth2-аутентификацию

**Ввод:**
```bash
asmo run "Реализовать OAuth2-аутентификацию"
```

**Анализ сложности:**
- Оценка: 75
- Уровень: Complex
- Категории: `security`, `feature`, `api`
- Модификаторы влияния: `security_impact`, `high_risk`

**Выбор workflow:** `feature_implementation_full`

**Назначенные агенты:**
1. **architect** -- проектирует OAuth2-поток (authorization code, PKCE), стратегию хранения токенов, архитектуру интеграции провайдеров
2. **security-specialist** -- ревьюит обработку токенов, CSRF-защиту, валидацию redirect URI, управление scope
3. **developer** -- реализует OAuth2-эндпоинты, обработчики callback, логику обновления токенов, интеграцию SDK провайдера
4. **tester** -- security-ориентированные тесты (истечение токена, невалидные гранты, CSRF-сценарии), интеграционные тесты с мок-провайдером

**Примерная длительность:** 4-6 часов

**YOLO-режим:** Нет -- сложная задача с последствиями для безопасности, требует явного одобрения.

**Ожидаемый результат:** 10+ файлов изменено/создано (маршруты аутентификации, middleware, сервис токенов, конфигурация провайдера, тесты), полный OAuth2 authorization code flow с PKCE, безопасное хранение токенов, механизм обновления и комплексные security-тесты.

## Задача 5: Спроектировать стратегию миграции на микросервисы

**Ввод:**
```bash
asmo run "Спроектировать стратегию миграции на микросервисы"
```

**Анализ сложности:**
- Оценка: 90
- Уровень: Enterprise
- Категории: `architecture`, `refactoring`
- Модификаторы влияния: `high_risk`

**Выбор workflow:** `architecture_design`

**Назначенные агенты:**
1. **architect** -- определяет границы сервисов, паттерны коммуникации (синхронные/асинхронные), владение данными, фазы миграции
2. **developer** -- оценивает осуществимость реализации, выявляет общий код и зависимости, предлагает порядок извлечения
3. **devops** -- проектирует стратегию деплоя, service mesh, мониторинг, изменения CI/CD-пайплайна
4. **security-specialist** -- ревьюит межсервисную аутентификацию, сетевые политики, управление секретами для распределённой системы

**Примерная длительность:** 8-12 часов

**YOLO-режим:** Нет -- enterprise-задача, требующая тщательного ревью и одобрения на каждой фазе.

**Ожидаемый результат:** Документы архитектурного дизайна, включающие карту границ сервисов, контракты коммуникации (спецификации API, схемы событий), план миграции с фазами и стратегиями отката, архитектуру деплоя, настройку мониторинга и наблюдаемости.

## Использование демо-данных для тестирования

Для верификации поведения ASMO по ожидаемым результатам:

```bash
# Быстрая проверка: совпадает ли рекомендация с ожидаемым workflow?
asmo suggest "Исправить опечатку в README"
asmo suggest "Реализовать OAuth2-аутентификацию"

# Анализ сложности: совпадает ли оценка с ожидаемым диапазоном?
asmo analyze "Добавить валидацию email в форму регистрации"
asmo analyze "Спроектировать стратегию миграции на микросервисы"

# Dry-run: совпадают ли назначенные агенты с ожиданиями?
asmo run "Исправить null pointer в обработчике логина" --dry-run
asmo run "Реализовать OAuth2-аутентификацию" --dry-run
```

Оценки могут варьироваться на несколько баллов в зависимости от LLM-режима (Session vs API vs Heuristics). Режим эвристик (~65% точности) может выдать другой выбор workflow для пограничных задач.
