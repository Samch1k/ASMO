# Известные проблемы и ограничения

Текущие ограничения и планируемые улучшения для ASMO v1.1.0.

---

## Текущие ограничения

### 1. Точность эвристик ~65%

Keyword-based fallback-режим (`--no-llm`) достигает примерно 65% точности для классификации задач и выбора workflow. Он может неверно классифицировать пограничные задачи, особенно когда ключевые слова баг-фиксов ("fix", "error") встречаются рядом с ключевыми словами фич ("add", "implement").

**Обходное решение:** Используйте Session- или API-режим для точной классификации. Если необходим режим эвристик, проверьте предложенный workflow через `asmo suggest` перед выполнением.

### 2. Session-режим требует Claude CLI

Режим Session по умолчанию ($0 стоимость) требует установленного и аутентифицированного Claude CLI. Если Claude CLI недоступен, ASMO переключается на API-режим, который требует `ANTHROPIC_API_KEY` и стоит за каждый токен.

**Обходное решение:** Установите Claude CLI (`brew install anthropic/claude/claude` на macOS) и выполните `claude auth login`. Альтернативно, задайте `ANTHROPIC_API_KEY` в `.env` для API-режима.

### 3. Отсутствие Web UI

ASMO на данный момент работает только в CLI. Нет веб-интерфейса для мониторинга прогресса workflow, ревью выходов агентов или управления конфигурациями.

**Обходное решение:** Используйте `--verbose` для детального вывода в терминале. Метрики хранятся в SQLite и могут быть запрошены напрямую.

### 4. Один LLM-вендор

ASMO на данный момент поддерживает только модели Anthropic (Claude через CLI или API). Нет интеграции с OpenAI, Google Gemini или другими LLM-провайдерами.

**Обходное решение:** На данный момент отсутствует. Claude от Anthropic -- единственный поддерживаемый бэкенд.

### 5. Только локальные метрики

Метрики и данные обучения хранятся в локальной SQLite. Нет облачной синхронизации, совместного использования командой или централизованного дашборда метрик.

**Обходное решение:** Базу SQLite можно вручную скопировать или забэкапить. Запрос стандартными инструментами SQLite.

### 6. Отсутствие DSL для кастомных агентов

Агенты определяются в TypeScript-коде внутри пакета `@asmo/core`. Нет декларативного DSL или формата конфигурации для определения кастомных агентов без написания кода.

**Обходное решение:** Расширяйте класс `BaseAgent` в TypeScript. Следуйте существующим паттернам агентов в `packages/core/src/agents/`.

### 7. Обратная связь при долгих workflow

Для сложных workflow (score > 70) выполнение может занять несколько часов. Обратная связь о прогрессе при длительных фазах ограничена verbose-выводом логов.

**Обходное решение:** Используйте `--verbose` для наблюдения за активностью агентов в реальном времени. Используйте `--dry-run` для превью ожидаемой длительности перед запуском.

## Дорожная карта

### v1.2.0 (Планируется)

- Улучшенная точность эвристик (цель: 80%+)
- Улучшенные сообщения об ошибках с действенными предложениями
- Workflow генерации кода из спецификации
- Расширенные категории keyword-анализа
- Улучшение трекинга и оценки времени выполнения workflow

### v2.0.0 (Планируется)

- Мульти-LLM поддержка (OpenAI, Google Gemini, локальные модели)
- Веб-дашборд мониторинга
- Облачные метрики с опциональной синхронизацией
- Фичи командной коллаборации
- Маркетплейс шаблонов workflow (внутренний)
- Система плагинов для кастомных интеграций

### v3.0.0 (Планируется)

- Самообучающиеся агенты (обучение на истории выполнения)
- Кастомный DSL для агентов (декларативные YAML/JSON-определения)
- Коммьюнити-маркетплейс для агентов, workflow и навыков
- Распределённое выполнение на нескольких машинах
- Продвинутая аналитика и отчётность

## Сообщение о проблемах

Для отчёта о баге или запроса фичи:

1. Запустите проблемную команду с `--verbose` для захвата детального вывода
2. Укажите версию ASMO (`asmo --version`)
3. Отметьте используемый LLM-режим (Session, API или Heuristics)
4. Предоставьте полную команду и все сообщения об ошибках
