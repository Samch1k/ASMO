# Демо-сценарий ASMO

Три управляемых сценария, демонстрирующих возможности ASMO. Каждый сценарий самодостаточен и может быть запущен независимо.

**Предварительные требования:**
- ASMO установлен (`pnpm install && pnpm build`)
- Claude CLI установлен и аутентифицирован (для Session-режима, $0 стоимость)
- Альтернативно: `ANTHROPIC_API_KEY` задан в `.env` (для API-режима)

---

## Сценарий 1: Простой баг-фикс (5 минут)

**Цель:** Продемонстрировать анализ задач ASMO, автоматический выбор workflow и YOLO-режим для тривиальных задач.

### Шаг 1: Проверить рекомендацию ASMO

```bash
asmo suggest "Исправить баг с регистрозависимым email при логине"
```

**Ожидаемый вывод:**
```json
{
  "useAsmo": true,
  "workflow": "bug_fix_workflow",
  "score": 35,
  "level": "simple",
  "reasoning": "Исправление бага с конкретной областью, затрагивает логику аутентификации"
}
```

**На что обратить внимание:** ASMO определяет это как баг-фикс (ключевые слова: "исправить", "баг") и выбирает `bug_fix_workflow`. Оценка ~35 указывает на простую задачу, которая выигрывает от мультиагентной оркестрации.

### Шаг 2: Выполнить с детальным выводом

```bash
asmo run "Исправить баг с регистрозависимым email при логине" --verbose
```

**Ожидаемое поведение:**
1. Агент debugger исследует проблему регистрозависимости
2. Агент developer реализует исправление (например, нормализация `.toLowerCase()`)
3. Агент tester валидирует исправление тест-кейсами

**На что обратить внимание:** В verbose-режиме наблюдайте вклад каждого агента. Debugger локализует проблему, developer создаёт точечное исправление, tester подтверждает корректность.

**Ожидаемый результат:** ~10 мин общее время, 2 файла изменены, все тесты пройдены.

### Шаг 3: Попробовать YOLO-режим с тривиальной задачей

```bash
asmo run "Исправить опечатку в README"
```

**Ожидаемое поведение:**
- Оценка сложности: ~15 (trivial)
- YOLO-режим активируется автоматически (score < 30)
- Нет чекпоинта одобрения -- задача выполняется немедленно
- Завершение < 60 секунд

**На что обратить внимание:** Задача полностью обходит шаг одобрения. YOLO-режим спроектирован для низкорисковых изменений, где ревью человеком не добавляет ценности.

## Сценарий 2: Реализация фичи (15 минут)

**Цель:** Продемонстрировать полный мультиагентный конвейер для фичи средней сложности, включая превью dry-run и координацию architect-developer-tester.

### Шаг 1: Анализ сложности

```bash
asmo analyze "Добавить страницу профиля пользователя с загрузкой аватара"
```

**Ожидаемый вывод:**
```json
{
  "score": 60,
  "level": "medium",
  "categories": ["feature", "api", "ui"],
  "suggestedWorkflow": "feature_implementation_full",
  "agents": ["architect", "developer", "tester"],
  "estimatedDuration": "4-6ч"
}
```

**На что обратить внимание:** Оценка отражает несколько аспектов: UI-компонент, API загрузки файлов, хранилище и тестирование. ASMO корректно определяет это как реализацию фичи, требующую полного конвейера.

### Шаг 2: Dry-run для превью плана

```bash
asmo run "Добавить страницу профиля пользователя с загрузкой аватара" --dry-run
```

**Ожидаемый вывод:**
```
Workflow: feature_implementation_full
Фазы:
  1. Архитектура    -> агент architect
  2. Разработка     -> агент developer
  3. Тестирование   -> агент tester

План:
  - architect: Определить структуру компонентов, API-контракты, стратегию хранения
  - developer: Реализовать страницу профиля, эндпоинт загрузки, обработку файлов
  - tester: Unit-тесты, интеграционные тесты, edge-кейсы загрузки
```

**На что обратить внимание:** Флаг `--dry-run` показывает план выполнения без запуска. Полезно для проверки назначений агентов и порядка фаз перед расходованием времени.

### Шаг 3: Выполнение workflow

```bash
asmo run "Добавить страницу профиля пользователя с загрузкой аватара"
```

**Ожидаемое поведение:**
1. **Фаза архитектуры:** Агент architect проектирует структуру компонентов, определяет API-контракты для загрузки аватара, выбирает стратегию хранения
2. **Фаза разработки:** Агент developer реализует страницу профиля, эндпоинт загрузки и логику обработки файлов
3. **Фаза тестирования:** Агент tester создаёт unit- и интеграционные тесты, покрывает edge-кейсы (размер файла, валидация формата)

**На что обратить внимание:**
- Context Cascade передаёт архитектурные решения developer (без повторной работы)
- Tester получает контекст и архитектуры, и реализации
- Каждая фаза строится на предыдущей

**Ожидаемый результат:** ~5ч, несколько файлов создано (компоненты, API-маршруты, тесты), API-контракты определены, тестовое покрытие сценариев загрузки.

## Сценарий 3: TEA-планирование (10 минут)

**Цель:** Продемонстрировать консолидированные TEA (Test Engineering & Automation) воркфлоу для структурированного планирования тестов.

### Шаг 1: Список доступных workflow

```bash
asmo workflow
```

**Ожидаемый вывод (фрагмент):**
```
Доступные воркфлоу (27):

Основная разработка:
  bug_fix_workflow            Исправление багов, исследование ошибок
  feature_implementation_full Новые фичи
  code_refactoring            Реструктуризация кода
  ...

TEA (Test Engineering & Automation):
  tea_planning_workflow       Планирование тестов: риски + стратегия + дизайн
  tea_execution_workflow      Выполнение тестов: автоматизация + регрессия
  tea_validation_workflow     Валидация качества: гейты + готовность к релизу
  ...
```

**На что обратить внимание:** 27 workflow организованы по категориям. TEA-воркфлоу консолидированы из исходных 8 в 3 фокусированных workflow.

### Шаг 2: Запуск TEA-планирования

```bash
asmo workflow tea_planning_workflow --task "Спланировать тесты для потока оформления заказа"
```

**Ожидаемое поведение:**
1. **Оценка рисков:** Выявление высокорисковых областей в оформлении заказа (обработка платежей, состояние корзины, конкурентность)
2. **Стратегия тестирования:** Определение подхода к тестированию (unit, integration, E2E), целей покрытия, инструментов
3. **Дизайн тестов:** Создание конкретных тест-кейсов, требований к данным, настройки окружения

**Ожидаемый вывод включает:**
- Матрицу рисков с рейтингами серьёзности и вероятности
- Документ стратегии тестирования с целями покрытия (например, 90% для логики платежей)
- Каталог тест-кейсов, организованный по уровню риска
- Оценка времени выполнения: 3-5 часов для фазы планирования

**На что обратить внимание:**
- Приоритизация по рискам: тесты платежей и управления состоянием ранжированы выше всего
- Конкретные, действенные тест-кейсы (а не просто категории)
- Чёткая прослеживаемость от риска к тест-кейсу

### Шаг 3: Превью workflow выполнения (опционально)

```bash
asmo run "Автоматизировать тесты потока оформления заказа" --dry-run
```

**Ожидаемый вывод:**
```
Workflow: tea_execution_workflow
Фазы:
  1. Автоматизация тестов  -> агент tester
  2. Регрессионный набор   -> агент tester
  3. План обслуживания     -> агент tester
```

**На что обратить внимание:** После планирования workflow выполнения подхватывает с того места, где остановилось планирование. Адаптивная детекция фаз может подключиться к workflow в середине, если некоторые фазы уже завершены.

---

## Советы для презентующих

- **Используйте `--verbose`** для показа пофазного прогресса агентов в реальном времени.
- **Используйте `--dry-run`** перед живым выполнением для установки ожиданий по длительности.
- **Начинайте со сценария 1**, чтобы продемонстрировать простейший путь, затем наращивайте сложность.
- **Выделите YOLO-режим** как ключевое отличие -- тривиальные задачи не требуют одобрения человека.
- **Выделите Context Cascade** в сценарии 2 -- каждый агент строит на работе предыдущего, контекст не теряется между фазами.
