## Что такое ASMO

ASMO (AI System for Multiagent Orchestration) -- это интеллектуальный фреймворк
оркестрации задач, который автоматически анализирует входящие задачи разработки,
определяет их сложность, выбирает наиболее подходящий воркфлоу и делегирует
выполнение специализированным AI-агентам. Вместо того чтобы разработчик вручную
выбирал нужный инструмент, процесс или агента, ASMO действует как
интеллектуальный диспетчер: читает описание задачи, оценивает, что необходимо
сделать, и формирует конвейер агентов, которые совместно доставляют полный
результат -- от архитектурного анализа через реализацию до тестирования и
валидации. Система реализует методологию BMAD (Breakthrough Method of Agile AI Driven
Development), где каждый воркфлоу привязан к реальному бизнес-результату.

ASMO построен как монорепозиторий с двумя пакетами: `@asmo/core` (библиотека
оркестрации, содержащая всех агентов, воркфлоу, навыки и логику движка) и
`@asmo/cli` (интерфейс командной строки, предоставляющий возможности ASMO
разработчикам). Система поддерживает стратегию Dual LLM, где режим по умолчанию
использует Claude CLI с нулевой стоимостью, с автоматическим откатом на Anthropic
API или эвристики на основе ключевых слов при необходимости. Это делает ASMO
практичным для повседневного использования -- сложные задачи получают полную
AI-оркестрацию, а простые задачи обрабатываются офлайн без API-вызовов.

## Проблема и решение

| Проблема | Решение ASMO |
|----------|-------------|
| Высокая стоимость LLM API-вызовов | Dual LLM: Session ($0) -> API -> Heuristics |
| Зависимости от внешних библиотек (Python, LangGraph, CrewAI) | Native TypeScript, `DynamicOrchestrator`, минимальные зависимости |
| Ручная маршрутизация задач между агентами | `TaskRouter` -- автоматический выбор модели и агента на основе `ComplexityAnalyzer` |
| Простые задачи блокируются одобрениями | YOLO Mode -- авто-одобрение для задач с complexity score < 30 |
| Разрозненная работа агентов, потеря контекста | Party Mode -- параллельная коллаборация с конвергенцией результатов |
| Разработчик сам решает, какой агент нужен | `ComplexityAnalyzer` + `WorkflowSelector` автоматически маршрутизируют задачу |
| Многошаговые задачи теряют контекст между фазами | `ContextCascade` распространяет артефакты и решения между фазами |
| Нет способа пропустить фазы при частичном выполнении | Adaptive Phase Detection определяет существующие артефакты и входит в воркфлоу с нужной фазы |
| Однобокие ревью пропускают уязвимости | Adversarial Review -- принудительный критический анализ с 3-уровневой эскалацией |
| Нет обратной связи для улучшения маршрутизации | Metrics & Learning Loop отслеживает результаты и корректирует выбор воркфлоу |
| Отказы агентов каскадируют и блокируют конвейер | `CircuitBreaker` изолирует отказы и обеспечивает graceful degradation |
| Непонятно, нужен ли ASMO для задачи | `asmo suggest` выдает мгновенную JSON-рекомендацию с обоснованием |

## Как это работает

Конвейер обработки ASMO состоит из пяти этапов -- от входящей задачи до
итогового результата:

```
                         Конвейер обработки ASMO
  ========================================================================

  [1] ВВОД ЗАДАЧИ             [2] АНАЛИЗ                  [3] МАРШРУТИЗАЦИЯ
  +-----------------+     +---------------------+    +-------------------+
  | "Исправить баг  |     | ComplexityAnalyzer  |    | WorkflowSelector  |
  |  авторизации,   | --> | - Анализ ключевых   | -> | - Выбор воркфлоу  |
  |  вызывающий     |     |   слов              |    | - Назначение      |
  |  ошибку 500     |     | - LLM-оценка        |    |   агентов         |
  |  на /api/auth"  |     | - Оценка: 35 (Simple)|   | - bug_fix_workflow|
  +-----------------+     +---------------------+    +-------------------+
                                                              |
                                                              v
  [5] РЕЗУЛЬТАТ                [4] ВЫПОЛНЕНИЕ
  +-----------------+     +--------------------------------------+
  | - Исправление   |     | WorkflowEngine                      |
  |   применено     | <-- | +----------------------------------+ |
  | - Тесты пройдены|     | | Фаза 1: debugger расследует      | |
  | - Артефакты     |     | | Фаза 2: developer реализует      | |
  | - Метрики       |     | | Фаза 3: tester валидирует        | |
  +-----------------+     | +----------------------------------+ |
                          | ContextCascade передает контекст     |
                          | CircuitBreaker защищает каждого агента|
                          | ApprovalCheckpoint контролирует       |
                          |   переходы между фазами              |
                          +--------------------------------------+
```

### Поэтапное описание

**Этап 1 -- Ввод задачи.** Разработчик предоставляет описание задачи на
естественном языке через CLI (`asmo run "<задача>"`). Ввод валидируется (длина,
кодировка, управляющие символы) и санитизируется перед обработкой.

**Этап 2 -- Анализ сложности.** `ComplexityAnalyzer` оценивает задачу, используя
до трех методов с откатом: (1) Session LLM через Claude CLI, (2) API LLM через
Anthropic API, (3) эвристики на основе ключевых слов. На выходе -- оценка
сложности (0-100), уровень сложности (trivial/simple/medium/complex/enterprise)
и обнаруженные категории (bug_fix, security, architecture и т.д.).

**Этап 3 -- Маршрутизация.** `WorkflowSelector` сопоставляет результат анализа
с лучшим воркфлоу из 27 доступных вариантов. `TaskRouter` валидирует выбор,
`SkillMatcher` разрешает необходимые навыки, `AgentRegistry` формирует команду
агентов.

**Этап 4 -- Выполнение.** `WorkflowEngine` оркестрирует многофазное выполнение.
Каждая фаза вызывает одного или нескольких агентов через `AgentExecutor`.
`ContextCascade` автоматически передает выходные данные ранних фаз в последующие.
`CircuitBreaker` защищает от отказов агентов. `ApprovalCheckpoint` контролирует
переходы между фазами (или автоматически одобряет в режиме YOLO).
`IterationManager` управляет логикой повторных попыток.

**Этап 5 -- Результат.** Итоговый вывод включает все артефакты, созданные
агентами (изменения кода, результаты тестов, документация и т.д.), метрики
выполнения и резюме. Метрики сохраняются `MetricsCollector` и передаются в
обучающий цикл `MetricsOptimizer` для улучшения будущей маршрутизации.

## Ключевые концепции

ASMO построен на четырех фундаментальных примитивах: Агенты, Воркфлоу, Навыки
и Роли.

### Агенты (25 шт.)

**Агент** -- это специализированная AI-персона с определенной ролью, набором
навыков и поведенческим профилем. Каждый агент наследует `BaseAgent`, который
предоставляет базовые методы: `callLLM()`, `callLLMForJSON()`, `createResult()`,
`createArtifact()` и `requestMCP()`.

Агенты разделены на три уровня:

| Уровень | Кол-во | Назначение |
|---------|--------|-----------|
| Core (Базовые) | 6 | Основные роли разработки (architect, developer, debugger, optimizer, tester, devops) |
| Specialized (Специализированные) | 15 | Эксперты по доменам (безопасность, UX, API, бизнес-анализ и т.д.) |
| Validation (Валидационные) | 4 | Контрольные точки качества (design-validator, merge-coordinator, post-deploy-monitor, requirements-validator) |

### Воркфлоу (27 шт.)

**Воркфлоу** -- это последовательность фаз, каждая из которых выполняется одним
или несколькими агентами в определенном порядке. Воркфлоу хранятся как JSON-шаблоны
и загружаются `ConfigLoader`. Каждый воркфлоу определяет:

- Последовательность фаз (какие агенты запускаются и в каком порядке)
- Зависимости между фазами (какие фазы должны завершиться до начала других)
- Требования к навыкам (какие навыки необходимы)
- Контрольные точки одобрения (где требуется одобрение человека)
- Ожидаемые артефакты (что каждая фаза должна произвести)

### Навыки (55 шт.)

**Навык** -- это сфокусированная способность, описанная в файле `SKILL.md`.
Навыки загружаются `SkillMDLoader` и сопоставляются с задачами `SkillMatcher`.
Каждый навык имеет название, описание, триггерные ключевые слова и инструкции
для агента. Расширенные маппинги зависимостей хранятся в
`templates/skills/skill-dependencies.json`.

### Роли (25 шт.)

**Роль** -- это поведенческий профиль, определенный в JSON. Роли задают, как
агент коммуницирует, какую позицию занимает и какие стандарты проводит. Роли
загружаются `RoleManager` и назначаются агентам при инициализации.

| Уровень | Кол-во | Примеры |
|---------|--------|---------|
| Core | 6 | architect, developer, debugger, optimizer, tester, devops |
| Specialized | 15 | security-specialist, ux-designer, product-owner, scrum-master, api-designer |
| Validation | 4 | design-validator, merge-coordinator, post-deploy-monitor, requirements-validator |

### Архитектурные блоки

**A. Анализ задач и маршрутизация** -- Принимает строку задачи, анализирует
сложность, выбирает подходящий воркфлоу, разрешает навыки и формирует команду
агентов. `ComplexityAnalyzer` выдает оценку и классификацию. `WorkflowSelector`
сопоставляет классификацию с шаблоном воркфлоу. `SkillMatcher` обогащает контекст
релевантными навыками. `AgentRegistry` предоставляет экземпляры агентов.

**B. Движок выполнения воркфлоу** -- Основной слой оркестрации. `WorkflowEngine`
(создается через фабричный метод `create()`) управляет жизненным циклом запуска
воркфлоу. `PhaseManager` отслеживает прогресс фаз. `IterationManager` управляет
повторами. `ApprovalCheckpoint` обеспечивает контрольные точки одобрения (с YOLO-
обходом для тривиальных задач). `AgentExecutor` вызывает агентов.
`CircuitBreaker` защищает от каскадных отказов. `ContextCascade` передает
контекст между фазами.

**C. Мультиагентная коллаборация** -- Поддерживает несколько паттернов
коллаборации. **Последовательный** -- по умолчанию: агенты выполняются друг за
другом с передачей контекста. **PartySession** обеспечивает параллельную
мультиагентную коллаборацию с конвергенцией. **BrainstormingSession** --
открытая идеация. **AdversarialReview** -- принудительный критический анализ
с обязательным обнаружением проблем.

**D. Адаптивная детекция фаз** -- LLM-анализ существующего контекста проекта
для определения, какие фазы воркфлоу можно пропустить. Если в проекте уже
есть архитектурная документация, система обнаруживает это и присоединяется к
воркфлоу на фазе реализации. `ContextAnalyzer` собирает состояние проекта,
`PhaseDetector` определяет точку входа. Метаданные детекции записываются в
состояние для обсервабилити.

**E. Метрики и обучающий цикл** -- Отслеживает метрики выполнения (время,
использование токенов, процент успеха, производительность агентов) и передает
их в обучающий цикл. `MetricsOptimizer` подключен к
`WorkflowEngine.initialize()` и корректирует будущие решения маршрутизации
на основе исторических данных.

**F. Конфигурация и шаблоны** -- Трехуровневая конфигурация с fallback.
`ConfigLoader` загружает определения ролей (JSON), определения навыков (файлы
SKILL.md) и шаблоны воркфлоу (JSON). `RoleManager` назначает поведенческие
профили. `PromptLoader` разрешает шаблоны промптов для агентов.

## Dual LLM Strategy

ASMO использует архитектуру двойного LLM-провайдера с автоматическим откатом
для минимизации стоимости при сохранении точности.

### Режимы провайдера

| Режим | Стоимость | Требования | Точность | Назначение |
|-------|-----------|------------|----------|-----------|
| **Session** (по умолчанию) | $0 | Claude CLI установлен и аутентифицирован | Высокая | По умолчанию для всех задач; экономично |
| **API** | Pay-per-token | `ANTHROPIC_API_KEY` в `.env` | Высокая | Откат, если CLI недоступен, или форсирование через `--use-api` |
| **Heuristics** | $0 | Ничего | ~65% | Офлайн-режим, форсирование через `--no-llm`, или когда оба LLM-провайдера недоступны |

### Цепочка отката

```
  Session Mode (Claude CLI, $0)
      |
      | недоступен
      v
  API Mode (ANTHROPIC_API_KEY, $$$)
      |
      | недоступен
      v
  Heuristics Mode (анализ ключевых слов, $0, ~65% точность)
```

### Подробности Session Mode

Session-режим вызывает Claude CLI (`claude -p "<промпт>"`) как подпроцесс.
Это задействует существующую подписку пользователя на Claude с нулевой
дополнительной стоимостью. Рекомендуемый режим для рабочих процессов разработки,
где Claude CLI уже установлен.

```bash
# Настройка Session mode
brew install anthropic/claude/claude    # macOS
curl -fsSL https://cli.anthropic.com/install.sh | sh  # Linux
claude auth login                        # Аутентификация
```

### Подробности API Mode

API-режим отправляет запросы напрямую в Anthropic API, используя переменную
окружения `ANTHROPIC_API_KEY`. Этот режим тарифицируется за токен, но не
требует установки Claude CLI.

```bash
# Настройка API mode
echo 'ANTHROPIC_API_KEY=sk-ant-...' > .env
```

### Выбор модели по сложности

Фабрика LLM-провайдера выбирает модель в зависимости от сложности задачи:

| Уровень сложности | Диапазон оценки | Модель | Обоснование |
|-------------------|-----------------|--------|-------------|
| Trivial | 0-20 | Haiku | Быстро, дешево, достаточно для простых задач |
| Simple | 21-40 | Haiku | Достаточно для прямолинейных исправлений |
| Medium | 41-60 | Sonnet | Сбалансированное рассуждение для стандартных фич |
| Complex | 61-80 | Sonnet/Opus | Глубокий анализ для безопасности, рефакторинга |
| Enterprise | 81-100 | Opus | Максимальные возможности для архитектуры, миграций |

Подробная техническая документация:
[LLM Provider Factory](../../packages/core/docs/llm-provider-factory.md).

## Ключевые фичи

### YOLO Mode

YOLO-режим позволяет тривиальным задачам автоматически обходить контрольные
точки одобрения. Когда оценка сложности задачи ниже 30, `ApprovalCheckpoint`
пропускает одобрение человека и переходит непосредственно к выполнению. Это
устраняет трение для простых задач -- исправление опечаток, текстовые правки,
мелкие патчи.

- **Порог:** complexity score < 30
- **Эффект:** авто-одобрение всех переходов между фазами
- **Отключение:** флаг `--no-yolo`

### Party Mode

Party Mode обеспечивает параллельную мультиагентную коллаборацию. Несколько
агентов работают над одной задачей одновременно, и их результаты объединяются
через этап конвергенции. Используется в воркфлоу, которые выигрывают от
множественных перспектив -- брейнштормы и комплексные ревью.

- **Паттерн:** параллельное выполнение с конвергенцией
- **Применение:** многоперспективный анализ, брейншторминг, комплексные ревью
- **Реализация:** классы `PartySession` и `BrainstormingSession`

### Adversarial Review

Воркфлоу Adversarial Review обеспечивает принудительный критический анализ,
требуя от агента `adversarial-reviewer` находить проблемы. В отличие от
стандартного код-ревью, adversarial review явно требует обнаружения проблем,
с трехуровневой эскалацией:

1. **Уровень 1:** Стандартный критический обзор -- code smells, потенциальные
   баги, проблемы дизайна
2. **Уровень 2:** Глубокий анализ -- уязвимости безопасности, узкие места
   производительности, проблемы масштабируемости
3. **Уровень 3:** Архитектурный обзор -- системные проблемы, изъяны проектирования,
   оценка технического долга

### Adaptive Phase Detection

Адаптивная детекция фаз использует LLM-анализ для определения, какие фазы
воркфлоу можно пропустить на основе существующих артефактов проекта. Например,
если проект уже имеет полную архитектурную документацию, система обнаруживает
это и присоединяется к воркфлоу на фазе реализации, а не начинает с
архитектурного проектирования.

- **Компоненты:** `ContextAnalyzer` (собирает состояние проекта),
  `PhaseDetector` (определяет точку входа)
- **Метаданные:** результаты детекции записываются в состояние воркфлоу
  для обсервабилити
- **Применение:** возобновление прерванных воркфлоу, подключение к текущим
  проектам

Подробная техническая документация:
[Adaptive Phase Detection](../../packages/core/docs/adaptive-phase-detection.md).

### Context Cascade

Context Cascade обеспечивает автоматическую передачу выходных данных ранних
фаз воркфлоу в последующие фазы как входной контекст. Когда агент `debugger`
создает отчет о расследовании на Фазе 1, этот отчет автоматически доступен
агенту `developer` на Фазе 2. Это устраняет ручную передачу контекста и
обеспечивает непрерывность конвейера.

- **Автоматически:** конфигурация не нужна; включено по умолчанию
- **Выборочно:** передаются только релевантные артефакты (не сырые логи)
- **Накопительно:** каждая фаза получает накопленный контекст всех предыдущих фаз

### BMAD-персонажи

ASMO включает четыре именованных профиля персонажей (BMAD), которые применяют
определенные философии разработки к агентам:

| Персонаж | Философия | Эффект |
|----------|----------|--------|
| **Amelia** | TDD-First | Агенты пишут тесты до реализации |
| **Winston** | Boring Technology | Агенты предпочитают проверенные стабильные решения новым |
| **Bob** | Zero Ambiguity | Агенты требуют точные спецификации, отвергают расплывчатые требования |
| **John** | WHY-First | Агенты задают вопрос о цели и мотивации перед началом работы |

## Справочник CLI

CLI `asmo` -- это основной интерфейс для взаимодействия с системой ASMO.
Распространяется как пакет `@asmo/cli` с точкой входа `packages/cli/bin/asmo.js`.

### Команды

#### `asmo run` -- Выполнить задачу

Главная команда. Анализирует задачу, выбирает воркфлоу и выполняет его.

```bash
# По умолчанию: Session mode -> API -> Heuristics
asmo run "Исправить баг авторизации, вызывающий ошибку 500 на /api/auth"

# Форсировать API mode (требуется ANTHROPIC_API_KEY)
asmo run "Реализовать интеграцию OAuth2" --use-api

# Отключить LLM, только эвристики (быстро, офлайн)
asmo run "Исправить опечатку в README" --no-llm

# Подробный вывод (детальное логирование)
asmo run "Рефакторинг платежного модуля" --verbose

# Пробный запуск (только анализ, без выполнения)
asmo run "Спроектировать микросервисную архитектуру" --dry-run

# Переопределить автоматический выбор воркфлоу
asmo run "Исправить регрессию логина" --workflow bug_fix_workflow
```

**Флаги:**

| Флаг | Описание |
|------|---------|
| `--use-api` | Форсирует API-режим (pay-per-token, требуется `ANTHROPIC_API_KEY`) |
| `--no-llm` | Отключает LLM-анализ, использует эвристики на ключевых словах (~65% точность) |
| `--verbose` | Детальный вывод логов |
| `--dry-run` | Только анализ, воркфлоу не выполняется |
| `--workflow <name>` | Переопределяет автоматический выбор воркфлоу конкретным воркфлоу |

#### `asmo suggest` -- Получить рекомендацию

Возвращает JSON-рекомендацию о том, стоит ли передавать задачу в ASMO.
Рассчитан на быструю оценку, подходит для хуков и автоматизации.

```bash
asmo suggest "Добавить валидацию ввода в форму регистрации"
```

**Пример вывода:**

```json
{
  "useAsmo": true,
  "confidence": 0.85,
  "reasoning": "Изменение нескольких файлов, затрагивает безопасность (валидация ввода), тестирование и логику форм",
  "suggestedWorkflow": "feature_implementation_full",
  "complexity": {
    "score": 55,
    "level": "medium"
  }
}
```

#### `asmo analyze` -- Анализ сложности

Возвращает детальный анализ сложности без выполнения воркфлоу.

```bash
asmo analyze "Мигрировать базу данных с PostgreSQL на MongoDB"
```

#### `asmo workflow` -- Управление воркфлоу

Просмотр доступных воркфлоу или выполнение конкретного.

```bash
# Показать все 27 воркфлоу
asmo workflow

# Выполнить конкретный воркфлоу с задачей
asmo workflow bug_fix_workflow --task "Исправить race condition в обработчике сессий"
asmo workflow security_audit --task "Проверить модуль аутентификации"
```

## Config Fallback Chain

ASMO использует трехуровневую систему конфигурации с каскадным откатом:

```
  Приоритет 1: .cursor/config
  (Среда Claude Code -- переопределения на уровне проекта)
      |
      | не найдено
      v
  Приоритет 2: ~/.asmo/config
  (Домашний каталог пользователя -- персональные настройки)
      |
      | не найдено
      v
  Приоритет 3: node_modules/@asmo/core/templates
  (Встроенные шаблоны -- всегда доступны, поставляются с пакетом)
```

### Содержимое каждого уровня

| Уровень | Расположение | Содержимое | Область переопределения |
|---------|-------------|-----------|------------------------|
| 1 (Проект) | `.cursor/config` | Переопределения для конкретного проекта | Высший приоритет, уровень проекта |
| 2 (Пользователь) | `~/.asmo/config` | Персональные настройки, пользовательские агенты | Уровень пользователя, для всех проектов |
| 3 (Встроенный) | `node_modules/@asmo/core/templates` | Все шаблоны по умолчанию, роли, навыки, воркфлоу | Базовые настройки, всегда доступны |

`ConfigManager` разрешает значения конфигурации, проверяя каждый уровень по порядку.
Первый уровень, предоставивший значение, побеждает. Это позволяет пользователям
переопределять встроенные настройки на уровне проекта или персональном уровне
без модификации самого пакета.

## Уровни сложности

ASMO классифицирует задачи по пяти уровням сложности на основе оценки анализа:

| Уровень | Диапазон | LLM-модель | Типичные задачи | Примеры воркфлоу |
|---------|----------|-----------|----------------|------------------|
| **Trivial** | 0-20 | Haiku | Опечатки, правки текста, обновление комментариев | Quick Flow, YOLO авто-одобрение |
| **Simple** | 21-40 | Haiku | Мелкие баг-фиксы, фичи в одном файле, правки конфигов | Bug Fix, Dev Story |
| **Medium** | 41-60 | Sonnet | Стандартные фичи, API-эндпоинты, умеренный рефакторинг | Feature Development, Code Review |
| **Complex** | 61-80 | Sonnet/Opus | Фичи безопасности, сложный рефакторинг, оптимизация | Security Audit, Performance Optimization |
| **Enterprise** | 81-100 | Opus | Редизайн архитектуры, миграции БД, системный дизайн | Architecture Design, Database Migration |

### Категории ключевых слов и оценки

Эвристический анализатор использует 14 категорий ключевых слов для офлайн-
классификации:

| Категория | Базовая оценка | Примеры ключевых слов |
|-----------|---------------|----------------------|
| `trivial` | 15 | typo, text, comment, rename |
| `bug_fix` | 25 | bug, fix, error, crash, regression |
| `refactoring` | 50 | refactor, cleanup, improve, restructure |
| `api` | 50 | endpoint, REST, GraphQL, route |
| `feature` | 55 | feature, implement, add, create |
| `database` | 60 | migration, schema, query, index |
| `performance` | 65 | optimize, speed, cache, memory, profiling |
| `security` | 70 | auth, encryption, oauth, vulnerability, XSS |
| `architecture` | 75 | architecture, system design, microservice |

**Модификаторы влияния** (добавляются к базовой оценке):

| Модификатор | Триггер | Эффект |
|-------------|---------|--------|
| `high_risk` | Ключевые слова: core, modify, redesign | +10 к оценке |
| `data_impact` | Обнаружены изменения, связанные с БД | Маркер чувствительности данных |
| `security_impact` | Обнаружены операции, связанные с безопасностью | Маркер ревью безопасности |
| `performance_impact` | Обнаружены критичные по производительности операции | Маркер ревью производительности |

## Быстрый старт

### Установка

```bash
# Установить CLI глобально
npm install -g @asmo/cli

# Или установить core-библиотеку для программного использования
npm install @asmo/core
```

### Настройка LLM-провайдера

**Вариант A: Session Mode (рекомендуется, стоимость $0)**

```bash
# Установить Claude CLI
brew install anthropic/claude/claude    # macOS
curl -fsSL https://cli.anthropic.com/install.sh | sh  # Linux

# Аутентификация
claude auth login
```

**Вариант B: API Mode (pay-per-token)**

```bash
# Создать файл .env в корне проекта
echo 'ANTHROPIC_API_KEY=sk-ant-...' > .env
```

### Первый запуск

```bash
# Проверить, стоит ли использовать ASMO для задачи
asmo suggest "Добавить rate limiting к API-эндпоинтам"

# Выполнить задачу (полный конвейер)
asmo run "Добавить rate limiting к API-эндпоинтам"

# Проанализировать сложность без выполнения
asmo analyze "Перепроектировать модуль аутентификации"

# Показать все доступные воркфлоу
asmo workflow

# Запустить с подробным выводом
asmo run "Исправить упавший тест логина" --verbose

# Пробный запуск для предпросмотра действий
asmo run "Реализовать слой кэширования" --dry-run
```

### Когда использовать ASMO, а когда нет

**Использовать ASMO, когда:**

- Необходимо изменить несколько файлов
- Затронуто несколько аспектов (код + тесты + документация)
- Требуются архитектурные решения
- Присутствуют сквозные аспекты (безопасность, производительность)
- Мультиагентная коллаборация была бы полезна (architect + developer + tester)

**Работать напрямую (без ASMO), когда:**

- Простые исправления опечаток
- Однострочные изменения
- Быстрые вопросы о кодовой базе
- Чтение и исследование файлов

### Быстрое решение

```bash
# Получить JSON-рекомендацию
asmo suggest "описание вашей задачи"
# Вернет: { useAsmo: true/false, reasoning: "...", suggestedWorkflow: "..." }
```

## Приложение: Валидация ввода

Все пользовательские промпты проходят валидацию через 10 проверок:

| Проверка | Описание |
|----------|---------|
| Проверка на пустоту/null | Отклоняет пустой или null-ввод |
| Минимальная длина | Требует минимум 3 символа |
| Максимальная длина | Ограничивает ввод 10 000 символами |
| Валидация UTF-8 | Проверяет корректную кодировку UTF-8 |
| Удаление управляющих символов | Удаляет null-байты и управляющие символы (безопасность) |
| Нормализация пробелов | Сворачивает избыточные пробелы |
| Обнаружение инъекций | Обнаруживает и санитизирует попытки инъекций в промпт |
| Нормализация Unicode | Нормализует Unicode в форму NFC |
| Фильтр контента | Базовая фильтрация содержимого |
| Валидация формата | Проверяет, что ввод является корректным описанием задачи |

## Приложение: Структура проекта

```
ASMO/
  packages/
    core/                    # @asmo/core - библиотека оркестрации
      src/
        agents/              # 25 реализаций агентов
        workflows/           # Движок воркфлоу и оркестрация
        skills/              # Сопоставление и загрузка навыков
        config/              # Управление конфигурацией
        metrics/             # Сбор метрик и обучение
        llm/                 # Фабрика LLM-провайдеров
      templates/
        roles/               # 25 определений ролей (JSON)
        skills/              # 55 определений навыков (SKILL.md)
        workflows/           # 27 шаблонов воркфлоу (JSON)
      docs/                  # Техническая документация
    cli/                     # @asmo/cli - интерфейс командной строки
      bin/
        asmo.js              # Точка входа CLI
      src/
        commands/            # Реализации команд CLI
  docs/
    en/                      # Документация на английском (5 фаз)
    ru/                      # Документация на русском (5 фаз)
    README.md                # Индекс документации
  CLAUDE.md                  # Интеграция с Claude Code
  CONTRIBUTING.md            # Руководство по вкладу
  CHANGELOG.md               # История версий
```

```bash
pnpm build

# Запустить тесты (jest)
pnpm test

# Запустить тесты с покрытием
pnpm test -- --coverage
```
