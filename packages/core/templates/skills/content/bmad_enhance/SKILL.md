# BMad Recommendation Enhancement

## Overview

Enhances existing BMad rule-based recommendations with deeper AI-powered analysis. This skill takes the basic recommendations generated by BMad's fallback mode and adds nuanced insights, context-aware suggestions, and implementation guidance.

## Quick Start

```bash
# Enhance recommendations for latest workflow
/bmad-enhance latest

# Enhance specific workflow recommendations
/bmad-enhance <workflow-id>

# Enhance with specific focus area
/bmad-enhance latest --focus <category>
```

## When to Use

Use this skill when:

- You have basic BMad recommendations and want deeper insights
- You see "💡 [BMad] Run: /bmad-enhance latest" in output
- Rule-based recommendations seem too generic
- You need specific implementation guidance
- Understanding trade-offs between optimization options

**Suitable for**: Workflows with existing basic recommendations

## What This Skill Does

1. **Reads Existing Recommendations**
   - Loads rule-based recommendations from retrospective report
   - Extracts workflow context and metrics
   - Identifies areas needing deeper analysis

2. **Enhances Each Recommendation**
   - Adds specific implementation steps
   - Explains trade-offs and risks
   - Provides code examples where relevant
   - Estimates actual impact based on context

3. **Prioritizes Intelligently**
   - Re-ranks by ROI (impact vs effort)
   - Considers project-specific constraints
   - Flags quick wins vs long-term improvements
   - Identifies dependencies between recommendations

4. **Updates Report**
   - Replaces basic recommendations with enhanced versions
   - Adds "AI Enhancement" section
   - Includes confidence scores for each recommendation

## Input Format

The skill expects:

- **`latest`** - Enhances most recent workflow's recommendations
- **`<workflow-id>`** - Enhances specific workflow by UUID
- **`--focus <category>`** - Optional focus (performance|quality|cost|process)

Category: Performance

Effort: Low

### Enhanced Recommendation (AI-Powered)
```
Priority: High
Category: Performance
Title: Reduce architect timeout from 30m to 5m
Description: Architect step completes in 2m on average (p95: 3m 20s) but has 30m timeout.
Current setting causes 27-minute delays when failures occur.

Enhancement:
  Reasoning: Analysis of 15 historical runs shows:
    - Average: 2m 10s
    - P95: 3m 20s
    - Max: 4m 5s
    Recommended timeout: 5m (150% of p95) provides safety margin while
    enabling 6x faster failure detection.

  Specific Steps:
    1. Open .cursor/config/orchestration/workflows.json
    2. Find "architect" step definition
    3. Change "timeout": "30m" to "timeout": "5m"
    4. Add "timeoutReasoning": "Based on p95 of 3m 20s + 50% buffer"
    5. Test with known-failure scenario to validate timeout works

  Code Example:
    {
      "order": 5,
      "role_id": "architect",
      "timeout": "5m",  // Changed from 30m
      "timeoutReasoning": "p95: 3m 20s + 50% = 5m"
    }

  Trade-offs:
    Pros:
      - 6x faster failure detection (5m vs 30m)
      - Saves 25 minutes per timeout failure
      - More responsive workflow experience
    Cons:
      - May timeout if architect requires deep analysis
      - Needs monitoring after deployment

  Impact Estimate:
    - Time Reduction: 25 minutes saved per timeout (rare but valuable)
    - Quality: No impact
    - Cost: Minimal (earlier failure = less compute)
    - Expected frequency: 1-2 timeouts per 50 runs

  Implementation:
    Files: .cursor/config/orchestration/workflows.json
    Changes:
      - Line 67: "timeout": "30m" → "timeout": "5m"
    Testing: Simulate architect hang with sleep() to verify timeout

  Risks:
    - Low: Legitimate long-running architect work might timeout
    - Mitigation: Monitor first 10 runs, adjust if needed

  Alternatives:
    - 10m timeout (more conservative, 3x improvement vs 6x)
    - Dynamic timeout based on task complexity
    - Parallel architect + timeout (run but don't wait)

Confidence: 0.92 (high - based on 15 historical data points)
Effort: Low (5-minute config change)
ROI: High (25 min saved × 2% failure rate = 30 sec/run average savings)
```
