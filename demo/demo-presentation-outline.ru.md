# ASMO Демонстрация для технической команды — План презентации

**Продолжительность:** 45-60 минут
**Аудитория:** Техническая команда (разработчики, архитекторы, технические лидеры)
**Цель:** Продемонстрировать возможности оркестрации мультиагентной системы ASMO, уникальные функции и потенциал интеграции

---

## 1. Введение (5 минут)

### Слайд: "Проблема"
**Текущие вызовы при AI-разработке:**
- 💰 Высокие затраты на API LLM → ограничивает внедрение
- 🎯 Ручная маршрутизация задач → накладные расходы разработчика
- 🔗 Потеря контекста между агентами → повторяющаяся работа
- ⏳ Усталость от подтверждений → простые задачи блокируются контрольными точками

### Слайд: "Решение ASMO"
**AI System for Multiagent Orchestration**

| Компонент | Значение |
|-----------|----------|
| Агенты | 25 (основные + специализированные + валидация) |
| Воркфлоу | 27 (от тривиальных до корпоративных) |
| Навыки | 55 (централизованная библиотека навыков) |
| Роли | 25 (поведенческие профили) |

**Ключевые отличия:**
- 💵 **Dual LLM**: Режим Session ($0) → API → Эвристики
- 🚀 **Режим YOLO**: Авто-подтверждение тривиальных задач (< 30 сложности)
- 🎭 **Party Mode**: Параллельное мультиагентное сотрудничество с консенсусом
- 🔄 **Адаптивное определение фазы**: Возобновление воркфлоу на любой фазе

## 2. Обзор архитектуры (10 минут)

### Слайд: "Конвейер ASMO — 5 этапов"

```
┌─────────────────────────────────────────────────────────────────┐
│ 1. ВВОД ЗАДАЧИ                                                  │
│    Разработчик: "Исправить null pointer в обработчике логина"   │
└─────────────────┬───────────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────────────────────────┐
│ 2. АНАЛИЗ СЛОЖНОСТИ (ComplexityAnalyzer)                        │
│    Session LLM → API LLM → Эвристики                            │
│    Вывод: {score: 35, level: "simple", categories: ["bug_fix"]} │
└─────────────────┬───────────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────────────────────────┐
│ 3. ВЫБОР ВОРКФЛОУ (WorkflowSelector)                            │
│    Соответствие: bug_fix_workflow                               │
│    Агенты: debugger → developer → tester                        │
└─────────────────┬───────────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────────────────────────┐
│ 4. ВЫПОЛНЕНИЕ (WorkflowEngine)                                  │
│    Фаза 1: debugger исследует                                   │
│    Фаза 2: developer исправляет (получает контекст из Фазы 1)   │
│    Фаза 3: tester проверяет (получает контекст из Фазы 1+2)     │
│    ContextCascade: Выходы фазы N → Входы фазы N+1               │
│    CircuitBreaker: защищает от сбоев агентов                    │
└─────────────────┬───────────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────────────────────────┐
│ 5. РЕЗУЛЬТАТ                                                     │
│    Артефакты: исправление применено, тесты добавлены            │
│    Метрики: {duration: "12m", tokensUsed: 4500, success: true}  │
│    MetricsOptimizer: учится на выполнении для будущей маршрутизации│
└─────────────────────────────────────────────────────────────────┘
```

### Слайд: "Стратегия Dual LLM — Оптимизация затрат"

| Режим | Стоимость | Требования | Точность | Сценарий использования |
|-------|-----------|------------|----------|------------------------|
| **Session** (по умолчанию) | $0 | Claude CLI установлен | Высокая | По умолчанию для всех задач |
| **API** (резервный) | За токен | ANTHROPIC_API_KEY | Высокая | Когда CLI недоступен или принудительно с `--use-api` |
| **Эвристики** (оффлайн) | $0 | Нет | ~65% | Оффлайн режим или принудительно с `--no-llm` |

**Цепочка автоматического резерва:**
```
Session ($0) → API ($$$) → Эвристики ($0, точность 65%)
```

**Выбор модели по сложности:**
- Тривиальные/Простые (0-40): Haiku
- Средние (41-60): Sonnet
- Сложные/Корпоративные (61-100): Sonnet/Opus

## 4. Глубокое погружение в уникальные функции (10 минут)

### Функция 1: Адаптивное определение фазы

**Проблема:** Начало воркфлоу с нуля, когда часть работы уже существует

**Решение:** LLM анализирует существующий контекст и присоединяется к воркфлоу на соответствующей фазе

**Пример:**
```bash
# Сценарий: Вы уже провели исследование, теперь нужно исправление + тесты

# 1. Создайте отчет об исследовании вручную
cat > investigation-report.md << EOF
# Исследование бага: Null Pointer в login.ts

Корневая причина: Отсутствует null проверка на объекте user в строке 42
Триггер: Объект user undefined когда сессия истекла
Исправление: Добавить null проверку перед обращением к user.email
EOF

# 2. Запустить bug_fix_workflow
asmo run "Fix null pointer in login handler" --verbose

# Ожидаемый вывод:
# [PhaseDetector] Анализирую существующий контекст...
# [PhaseDetector] Найдено: investigation-report.md
# [PhaseDetector] Решение: Присоединиться к воркфлоу на Фазе 2 (developer)
# [PhaseDetector] Пропускаю Фазу 1 (debugger) - уже завершена
# [PhaseDetector] Метаданные: {detectedPhase: 2, skippedPhases: [1]}
#
# [WorkflowEngine] Начинаю с Фазы 2...
# [Developer] Читаю отчет об исследовании...
# [Developer] Реализую исправление на основе находок...
```

**Преимущества:**
- ⏭️ Пропуск ненужных фаз
- 💰 Экономия вызовов LLM
- ⏱️ Снижение времени выполнения
- 🔄 Возобновление прерванной работы

**Сценарии использования:**
- Возобновление после прерывания
- Присоединение к существующему проекту в процессе
- Работа уже частично выполнена

### Функция 2: Adversarial Review

**Проблема:** Стандартные code review могут пропускать критические проблемы (предвзятость подтверждения)

**Решение:** Принудить агента найти проблемы с 3-уровневой эскалацией

**Конфигурация:**
```typescript
adversarial_review_workflow:
  agent: adversarial-reviewer
  mode: MUST_FIND_ISSUES
  escalation_levels: 3

  Уровень 1: Стандартный критический обзор
    - Код смеллы
    - Потенциальные баги
    - Проблемы дизайна

  Уровень 2: Глубокий анализ (если L1 находит < 3 проблем)
    - Уязвимости безопасности
    - Узкие места производительности
    - Проблемы масштабируемости

  Уровень 3: Архитектурный обзор (если L2 находит < 3 проблем)
    - Недостатки системного дизайна
    - Оценка технического долга
    - Долгосрочная поддерживаемость
```

**Пример:**
```bash
asmo workflow adversarial_review_workflow \
  --task "Review authentication module for production deployment"

# Ожидаемый вывод:
# [AdversarialReviewer] Уровень 1: Стандартный обзор
# ✗ Найдено 2 проблемы (ниже порога 3) → Эскалация на Уровень 2
#
# [AdversarialReviewer] Уровень 2: Глубокое погружение в безопасность
# ✓ Найдено 5 проблем:
#   1. [КРИТИЧНО] Хеширование паролей использует устаревшие раунды bcrypt (10 → должно быть 12+)
#   2. [ВЫСОКИЙ] Токены сессии не ротируются при повышении привилегий
#   3. [ВЫСОКИЙ] Отсутствует rate limiting на endpoint логина (риск brute force)
#   4. [СРЕДНИЙ] Сообщения об ошибках раскрывают информацию о существовании пользователя
#   5. [НИЗКИЙ] Возможна timing атака на сравнении пароля
```

**Сценарии использования:**
- Pre-production обзор безопасности
- Критические изменения инфраструктуры
- Требования соответствия (PCI-DSS, SOC2)

### Функция 3: Метрики и цикл обучения

**Как это работает:**
```typescript
// После каждого выполнения:
1. MetricsCollector записывает:
   - Использованный воркфлоу
   - Оценка сложности
   - Продолжительность
   - Использование токенов
   - Успех/неудача
   - Производительность агента

2. MetricsPersister сохраняет в SQLite:
   - Исторические данные
   - Анализ трендов

3. MetricsOptimizer учится:
   - Какие воркфлоу работают лучше для каких диапазонов сложности

   - Показатели успеха агентов
   - Эффективность токенов

4. Будущая маршрутизация использует изученные данные:
   - "bug_fix_workflow работает хорошо для оценки 30-40"
   - "feature_implementation_full: среднее 4.5ч для оценки 50-60"
```

```bash
#
# Рекомендация: Оптимально для простых исправлений багов (оценка 21-40)
```

**Преимущества:**
- 📈 Непрерывное улучшение
- 🎯 Лучший выбор воркфлоу со временем
- ⏱️ Точные оценки времени
- 💰 Оптимизация использования токенов

### Функция 4: Глубокое погружение в Party Mode

**Алгоритм конвергенции:**
```typescript
interface PartyModeConfig {
  maxRounds: 5              // Максимум раундов обсуждения
  convergenceThreshold: 0.8  // Требуется 80% согласия
  minAgents: 2              // Минимум агентов для party mode
}

// После каждого раунда:
function checkConvergence(round: number) {
  const similarity = calculateSimilarity(
    outputs[round],
    outputs[round - 1]
  )

  if (similarity >= threshold) {
    // Консенсус достигнут
    return consolidateResults(outputs[round])
  }

  if (round >= maxRounds) {
    // Достигнут максимум раундов, используем лучший результат
    return selectBestResult(outputs)
  }

  // Продолжаем следующий раунд с уточненными промптами
  return continueDiscussion(round + 1)
}
```

**Расчет схожести:**
```typescript
// Сравнивает выводы агентов используя:
1. Семантическая схожесть (расстояние эмбеддингов)
2. Структурная схожесть (перекрытие JSON схемы)
3. Схожесть решений (совпадение ключевых решений)

// Пример:
Агент A: "Использовать OAuth2 с PKCE"
Агент B: "Реализовать OAuth2 Authorization Code flow с расширением PKCE"
Агент C: "Использовать OAuth2 authorization code + PKCE для мобильной безопасности"

→ Семантическая схожесть: 0.92 (очень высокая)
→ Конвергенция: ✅ (выше порога 0.8)
```

**Анализ затрат на токены:**
```
Последовательный режим (3 агента):
  1 раунд × 3 агента × 1.5K токенов = 4.5K токенов

Party mode (3 агента, 3 раунда):
  3 раунда × 3 агента × 1.5K токенов = 13.5K токенов

Компромисс:
  + Лучшее качество (множественные перспективы)
  + Снижение слепых зон
  - 3x затраты на токены

→ Используйте для сложных задач (≥60), где качество имеет значение
```

```bash
asmo analyze "<задача>"

# Предпросмотр плана выполнения
asmo run "<задача>" --dry-run

# Выполнение с опциями
asmo run "<задача>" --verbose --use-api --workflow custom_workflow
```

**Флаги:**
- `--use-api`: Принудительный режим API (когда Session недоступен)
- `--no-llm`: Оффлайн режим эвристик (~65% точности)
- `--verbose`: Подробные логи активности агентов
- `--dry-run`: Предпросмотр без выполнения
- `--workflow <name>`: Переопределить выбор воркфлоу

### Пример интеграции Git Hook

```bash
# .git/hooks/pre-commit
#!/bin/bash

# Получить список измененных файлов
CHANGED_FILES=$(git diff --cached --name-only)

# Спросить ASMO нужна ли проверка
SUGGESTION=$(asmo suggest "Review changes: $CHANGED_FILES")
USE_ASMO=$(echo "$SUGGESTION" | jq -r '.useAsmo')

if [ "$USE_ASMO" = "true" ]; then
  echo "🤖 ASMO рекомендует оркестрированную проверку для этих изменений"
  echo "$SUGGESTION" | jq -r '.reasoning'

  read -p "Запустить ASMO review? (y/n) " -n 1 -r
  echo

  if [[ $REPLY =~ ^[Yy]$ ]]; then
    asmo workflow code_review_workflow \
      --task "Review staged changes: $CHANGED_FILES"
  fi
fi
```

## 7. Темы для вопросов и ответов

**Технические вопросы:**
1. Как ContextCascade обрабатывает большие артефакты? (Ответ: Выборочное распространение, только релевантные данные)
2. Что происходит при сбое агента? (Ответ: CircuitBreaker изолирует сбой, плавная деградация)
3. Можем ли мы добавить кастомных агентов? (Ответ: Да, расширяет BaseAgent, определите JSON роли + навыки)
4. Насколько точен режим эвристик? (Ответ: ~65%, хорошо для оффлайн/быстрых проверок)
5. Каково использование токенов для средней задачи? (Ответ: Простые: 2-4K, Средние: 5-10K, Сложные: 15-20K)

**Вопросы интеграции:**
1. Может ли ASMO интегрироваться с существующим CI/CD? (Ответ: Да, через CLI + exit коды)
2. Работает ли с моделями не от Claude? (Ответ: Сейчас ориентирован на Claude, расширяемая архитектура)
3. Можем ли мы кастомизировать воркфлоу? (Ответ: Да, JSON шаблоны в `~/.asmo/config/workflows/`)
4. Поддерживает ли язык нашей кодовой базы (Java/Go/Rust)? (Ответ: Да, не зависит от языка)

**Вопросы стоимости/производительности:**
1. Что если Claude CLI недоступен? (Ответ: Автоматический переход на режим API)
2. Можем ли мы ограничить использование токенов? (Ответ: Да, настройте максимум раундов для party mode)
3. Есть ли способ предпросмотра стоимости перед выполнением? (Ответ: Да, `--dry-run` показывает план агентов)

## Заметки для ведущего

**Точки акцента:**
1. **Стоимость $0** с режимом Session — прорыв для внедрения
2. **Режим YOLO** устраняет усталость от подтверждений (20% задач)
3. **Context Cascade** автоматический, не ручной (победа пользовательского опыта)
4. **Party Mode** показывает интеллект, а не просто последовательное выполнение
5. **Адаптивное определение фазы** показывает, что система умная, не жесткая

**Потенциальные возражения и ответы:**
- *"Слишком сложно для простых задач"*: Покажите режим YOLO (< 1 мин для тривиальных)
- *"Что если LLM ошибется?"*: Покажите ApprovalCheckpoint для оценки > 30
- *"Затраты на токены слишком высоки"*: Покажите режим Session ($0) vs сравнение API
- *"У нас уже есть инструменты"*: Подчеркните оркестрацию мультиагентов, не одного агента
- *"Привязка к Claude?"*: Упомяните, что архитектура расширяема

**План резервной демонстрации:**
Если живая демонстрация не работает:
1. Используйте предварительно записанные видеоклипы
2. Покажите примеры выходных файлов из предыдущих запусков
